Preguntas de Backend — Todo lo de las Fases 1–4
SOLID con ejemplos de código real, DI y su relación con testabilidad, REST vs alternativas, CQRS en la práctica, async/await correctamente explicado, y cómo responder preguntas de arquitectura bajo presión
⏱ ~5h
En una entrevista técnica de backend, el entrevistador no solo evalúa si conoces la respuesta — evalúa cómo razonas cuando no la sabes al 100%. La técnica correcta: empezar con la definición limpia, dar un ejemplo de código o arquitectura real, mencionar el trade-off o el cuándo NO usarlo, y conectar con algo que hayas construido. "SOLID es un conjunto de cinco principios de diseño de software orientado a objetos. En mi proyecto, el que más apliqué fue el Single Responsibility Principle al extraer la lógica de validación en clases separadas usando FluentValidation — en lugar de tener esa lógica en los controladores. El trade-off es más archivos, pero cada clase tiene una sola razón para cambiar."
Easy Explica SOLID y da un ejemplo de S y D de tu código ›
S — Single Responsibility: cada clase tiene una sola razón para cambiar. Ejemplo: OrderValidator solo valida, OrderRepository solo persiste, OrderService solo orquesta. En mi proyecto separé la validación con FluentValidation en lugar de mezclarla en los controladores.

D — Dependency Inversion: las capas de alto nivel no dependen de las de bajo nivel — ambas dependen de abstracciones. En la práctica: el OrderService depende de IOrderRepository (interfaz), no de SqlOrderRepository (implementación concreta). Esto permite testear con mocks y cambiar la implementación sin tocar el servicio.
Med ¿Cuándo usarías CQRS y cuándo es over-engineering? ›
CQRS (Command Query Responsibility Segregation) separa las operaciones de escritura (Commands) de las de lectura (Queries). Cuándo usarlo: cuando las lecturas y escrituras tienen requisitos muy distintos de escala o complejidad, cuando hay múltiples read models del mismo dato, o cuando usas Event Sourcing. En mi proyecto lo usé con MediatR: los Commands validan y mutan estado, las Queries son optimizadas con proyecciones directas a DTOs sin pasar por el domain model.

Cuándo es over-engineering: en CRUD simples con pocos usuarios, cuando el equipo es pequeño y la complejidad añadida no justifica el beneficio, o en early-stage donde los requisitos cambian constantemente. La regla: si tus lecturas y escrituras tienen el mismo modelo de datos y escala similar, un simple Repository es suficiente.
Med Explica async/await en .NET — ¿qué pasa realmente en el thread pool? ›
async/await no crea threads nuevos — libera el thread actual mientras espera operaciones I/O para que pueda atender otras requests. Cuando escribes await _db.GetAsync(), el thread del request es devuelto al thread pool mientras la base de datos procesa. Cuando la BD responde, el runtime toma un thread disponible (puede ser el mismo u otro) y continúa la ejecución.

Error común a mencionar: async void (solo para event handlers — no se puede await, las excepciones se pierden), .Result o .Wait() que causan deadlocks en ASP.NET classic, y no usar ConfigureAwait(false) en librerías. En ASP.NET Core, ConfigureAwait es menos crítico porque no hay SynchronizationContext, pero es buena práctica en código de librería.
Hard ¿Cuál es la diferencia entre el Outbox Pattern y usar una cola directamente? ›
El problema: si publicas un evento a RabbitMQ/Service Bus después de hacer el SaveChanges() en la base de datos, hay una ventana donde el DB save tiene éxito pero la publicación del evento falla — el sistema queda en estado inconsistente.

Outbox Pattern: en la misma transacción que guardas los datos, escribes el evento en una tabla OutboxMessages en la misma BD. Un proceso background separado (o un job) lee esa tabla y publica a la cola, marcando el mensaje como procesado. Si la publicación falla, reintenta — si tiene éxito, no hay inconsistencia posible.

vs. Cola directa: la cola directa es "at most once delivery" — puede perder mensajes. El Outbox garantiza "at least once delivery". La contrapartida: el consumer debe ser idempotente porque puede recibir el mismo mensaje más de una vez. En mi proyecto lo implementé con MassTransit que tiene soporte nativo de Outbox con EF Core.

SQL Avanzado en Formato Entrevista
Window functions vs GROUP BY, execution plan y cómo optimizar una query lenta, índices covering vs composite, diferencias entre niveles de aislamiento, y cómo diagnosticar un deadlock
⏱ ~5h
Las preguntas de SQL en entrevistas senior tienen un patrón: el entrevistador muestra una query o un problema de performance y te pide diagnosticar o mejorar. El enfoque correcto no es memorizar respuestas — es tener un framework de diagnóstico. Para una query lenta: primero revisar el execution plan (¿hay table scans en tablas grandes?), luego verificar los índices (¿la WHERE clause usa una columna indexada?), luego revisar estadísticas desactualizadas, y finalmente considerar si la query puede reestructurarse. Tener este proceso en la cabeza permite responder incluso preguntas que no has visto antes.
Med ¿Cuándo usas ROW_NUMBER() vs RANK() vs DENSE_RANK()? ›
Las tres son window functions que asignan números de fila, pero manejan empates distinto:

ROW_NUMBER(): número único siempre, sin importar empates. Si dos filas tienen el mismo valor, una recibe 1 y otra 2 — arbitrariamente. Útil para paginación y deduplicación.

RANK(): si hay empate en posición 2, ambas filas reciben 2, y la siguiente recibe 4 (salta el 3). Útil para rankings donde el "salto" es semánticamente correcto (como clasificaciones deportivas).

DENSE_RANK(): si hay empate en posición 2, ambas filas reciben 2, y la siguiente recibe 3 — sin saltos. Útil cuando necesitas rankings continuos sin huecos.

Ejemplo de entrevista: "Dame el segundo salario más alto por departamento" — DENSE_RANK() OVER (PARTITION BY dept ORDER BY salary DESC) con WHERE rnk = 2.
Hard Un query tarda 8 segundos. ¿Cómo lo diagnosticas y optimizas? ›
Paso 1 — Execution plan: SET STATISTICS IO ON; SET STATISTICS TIME ON; antes del query, o ver el Actual Execution Plan en SSMS. Buscar: Table Scan en tablas grandes (señal de índice faltante), Key Lookup caro (señal de índice non-covering), Sort o Hash Match en tablas grandes, flechas gruesas (muchas filas pasando entre operadores).

Paso 2 — Índices: verificar que las columnas en WHERE, JOIN y ORDER BY tienen índices. Si hay Key Lookup, considerar un covering index que incluya las columnas del SELECT con INCLUDE().

Paso 3 — Estadísticas: UPDATE STATISTICS [tabla] — estadísticas desactualizadas hacen que el query optimizer tome decisiones malas.

Paso 4 — Query rewrite: ¿hay subqueries que pueden ser JOINs? ¿Funciones en la WHERE clause que rompen el SARGability (ej: WHERE YEAR(fecha) = 2024 vs WHERE fecha BETWEEN '2024-01-01' AND '2024-12-31')?

Paso 5 — Concurrencia: si el problema es intermitente, puede ser bloqueo o deadlock — revisar sys.dm_exec_requests.
Hard Explica ACID y qué pasa en cada nivel de aislamiento ›
ACID: Atomicity (todo o nada), Consistency (las reglas de integridad se mantienen), Isolation (transacciones concurrentes no se interfieren), Durability (los commits sobreviven fallos).

Problemas de concurrencia: Dirty Read (leer datos no commiteados), Non-repeatable Read (la misma fila devuelve valores distintos en la misma tx), Phantom Read (nuevas filas aparecen en lecturas repetidas).

READ UNCOMMITTED: ve dirty reads — no se usa en producción salvo para reportes aproximados. READ COMMITTED (default de SQL Server): elimina dirty reads, permite non-repeatable reads. REPEATABLE READ: elimina dirty y non-repeatable reads, permite phantom reads. SERIALIZABLE: elimina todos los problemas — mayor consistencia, menor concurrencia, riesgo de deadlocks. SNAPSHOT: el equilibrio de SQL Server — lecturas ven una copia point-in-time, writes siguen bloqueándose entre sí. Excelente para reads de alta concurrencia.

System Design — Diseñar sistemas a escala
El framework RADDS para responder en 45 minutos, los 5 problemas más frecuentes (URL shortener, rate limiter, notification system, order processing, chat), y cómo conectar el diseño con lo que construiste
⏱ ~5h
Las entrevistas de system design no tienen una respuesta correcta única — evalúan tu proceso de razonamiento, no la solución final. El error más común: empezar a diagramar componentes inmediatamente sin clarificar los requisitos. El entrevistador quiere ver cómo navegas la ambigüedad, cómo priorizas requisitos funcionales vs no-funcionales, cómo justificas trade-offs. Un buen framework de 45 minutos: 5 min clarificando requisitos y escala, 5 min estimaciones de carga, 15 min diseño de alto nivel, 15 min deep dive en el componente más crítico, 5 min bottlenecks y mejoras. Tener este esquema memorizado elimina el pánico al inicio.

Live Coding — Algorítmica y LINQ bajo presión
El proceso de "think aloud" durante live coding, los 10 patrones algorítmicos que aparecen el 80% del tiempo, LINQ en entrevistas .NET, y cómo manejar el bloqueo mental
⏱ ~5h
El live coding evalúa tres cosas distintas: si llegas a la solución, cómo comunicas el proceso, y cómo respondes bajo presión. La comunicación vale tanto como la solución. Antes de escribir una sola línea: leer el problema en voz alta, preguntar por edge cases (¿puede ser null? ¿array vacío?), proponer el approach antes de implementarlo ("creo que puedo resolver esto con un sliding window — ¿te parece bien si empiezo por ahí?"). Si te bloqueas: verbalizar dónde estás bloqueado ("no recuerdo la sintaxis exacta de GroupBy con múltiples keys, pero la lógica sería..."). El entrevistador puede ayudarte si sabe dónde estás.

Simulación Completa — La entrevista real
Behavioral questions con el método STAR, cómo negociar el salario, simulación cronometrada de 45 min con IA o en Pramp, y la mentalidad correcta para el día de la entrevista
⏱ ~4h
Las behavioral questions no son relleno — son tan importantes como las técnicas en empresas maduras. El método STAR (Situation, Task, Action, Result) es la estructura que los entrevistadores esperan: describe la situación con suficiente contexto, cuál era tu responsabilidad específica, qué acciones tomaste tú (no "el equipo"), y cuál fue el resultado medible. Las preguntas más frecuentes que debes preparar: "Tell me about a time you disagreed with a technical decision", "Describe a project you're most proud of", "Tell me about a time you failed", y "How do you handle working with unclear requirements". El error clásico: dar respuestas genéricas sin métricas ni especificidad.
Pregunta Lo que evalúan Respuesta modelo
Tell me about your most complex project Profundidad técnica, decisiones de arquitectura, capacidad de comunicar complejidad → Proyecto del roadmap: 6 microservicios, Azure Container Apps, CI/CD, OAuth, observabilidad. Mencionar 2–3 decisiones técnicas específicas con sus trade-offs y las métricas resultantes.
Tell me about a technical decision you disagreed with Comunicación, madurez profesional, capacidad de defender posición con datos → Estructura: S: contexto técnico → T: tu rol → A: cómo presentaste tu perspectiva con datos, cómo escuchaste la otra posición → R: la decisión tomada y qué aprendiste. No importa si "ganaste" o no.
Where do you see yourself in 3 years? Ambición calibrada, alineación con el crecimiento de la empresa, autoreflexión → Respuesta honesta: profundizar en arquitectura distribuida, liderar técnicamente un equipo pequeño, contribuir a decisiones de diseño de sistemas. Conectar con lo que esa empresa específica puede ofrecer.
