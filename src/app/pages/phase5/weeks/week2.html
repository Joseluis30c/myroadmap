Escribir bullets de impacto — La habilidad central del CV
Las 3 fórmulas de Laszlo Bock (ex VP Google), verbos de acción para roles técnicos, cómo encontrar métricas reales en tu propio trabajo, y los errores que hacen que los CVs sean descartados en 10 segundos
⏱ ~4h
El 90% de los CVs técnicos tienen bullets que describen responsabilidades, no logros. "Responsible for maintaining the API" describe un rol. "Reduced API error rate from 3.2% to 0.4% by implementing structured exception handling and circuit breakers" describe un logro. La diferencia es una métrica y un verbo de acción fuerte. Si nunca has medido el impacto de tu trabajo, este es el momento: Application Insights tiene los datos de las últimas semanas del roadmap — latencia P99 antes/después del cache, tasa de errores, tiempo de deployment. Esos números reales son tu materia prima.
⚠️ Los 5 errores que hacen que un CV técnico sea descartado en 10 segundos: (1) Lista de tecnologías sin contexto — "Skills: C#, .NET, SQL, Azure" sin explicar cómo las usaste. (2) Bullets en primera persona: "I developed..." — nunca se usa "I" en un CV. (3) Soft skills sin evidencia: "Team player, fast learner" — no dice nada. (4) Responsabilidades sin resultados: "Maintained REST APIs" vs "Optimized 3 high-traffic endpoints, reducing p99 latency by 65%". (5) CV de más de 2 páginas para menos de 7 años de experiencia.
💡 Si no tienes métricas, créalas hoy. Abre Application Insights del proyecto del roadmap y documenta: latencia P99 del endpoint más usado, tasa de errores, tiempo de build/deploy en GitHub Actions, número de servicios desplegados, cobertura de tests. Estos números reales — aunque vengan de un proyecto personal — son 10 veces más persuasivos que una descripción genérica.

Construir el CV Técnico — Estructura y formato
El formato ATS-friendly que usan los recruiters de empresas tech, cómo presentar el proyecto del roadmap como experiencia real, la sección de Skills con el stack completo, y cómo adaptar el CV por rol
⏱ ~5h
El 75% de los CVs nunca llega a ojos humanos — son filtrados por sistemas ATS (Applicant Tracking Systems) antes. Un CV ATS-friendly tiene: texto puro sin tablas ni columnas complejas (los ATS no leen layouts de dos columnas bien), keywords exactas de la oferta de trabajo (si dice ".NET 8" incluye ".NET 8", no solo ".NET"), formato de fecha consistente (Jan 2024 – Present), y verbos de acción en pasado para experiencias anteriores y presente para el rol actual. El formato estándar que pasa todos los filtros: nombre + contacto → resumen técnico (3 líneas) → experiencia (más reciente primero) → proyectos → educación → skills.

GitHub Profesional — El portafolio que habla solo
Profile README que impacta en los primeros 5 segundos, repositorios del roadmap con documentación real, pinned repos estratégicos, contribution graph activo, y el README del proyecto principal
⏱ ~5h
Para un developer, GitHub es más importante que el CV. Un recruiter técnico o un CTO mira el GitHub antes de leer el CV completo. Lo que evalúan en los primeros 30 segundos: (1) el contribution graph — ¿commit frecuente o actividad de un día?, (2) los repos pinneados — ¿hay proyectos reales con README?, (3) la calidad de los READMEs — ¿explica qué hace el proyecto, cómo ejecutarlo, y las decisiones técnicas?, (4) la calidad de los commits — ¿son descriptivos o dicen "fix" y "update"? El proyecto del roadmap tiene todo lo necesario para un GitHub de nivel senior — solo requiere documentación y presentación.

LinkedIn Optimizado para el Mercado Tech
El headline que aparece en búsquedas, el About como carta de presentación técnica, la sección Experience con los bullets del Día 1, el Featured con el proyecto cloud, y cómo activar "Open to Work" para recruiters
⏱ ~4h
LinkedIn tiene un motor de búsqueda interno — el algoritmo decide qué perfiles aparecen cuando un recruiter busca ".NET developer" o "backend developer Azure". Las variables que más pesan: (1) el headline (las 220 caracteres debajo del nombre — no el título del cargo, sino las keywords del rol que buscas), (2) las primeras 3 líneas del About (lo único visible sin hacer clic en "ver más"), (3) el número de endorsements en skills específicos, y (4) la actividad reciente. El error más común: headline que dice el cargo actual ("Backend Developer at Empresa X") en lugar de las skills que los recruiters buscan ("Backend .NET 8 | Microservices | Azure | C# | REST APIs | Open to remote").

Perfil completo y revisión final
Revisar el CV con la fórmula XYZ, verificar que el GitHub project README funciona en 3 comandos, ajustar el LinkedIn para SSI (Social Selling Index) alto, y preparar el cover letter técnico en inglés
⏱ ~4h
El cover letter técnico es opcional en muchas empresas — pero cuando se pide, es la diferencia. Un cover letter de developer no debe ser una carta formal de presentación — debe ser una respuesta directa a "¿por qué tú para este rol específico?". Tres párrafos: (1) una oración de conexión directa entre tus skills y el rol ("I've spent the last year building production microservices on Azure — which aligns directly with what you're looking for"), (2) el logro más relevante con métrica, y (3) por qué esta empresa específicamente. Nunca copiar y pegar el mismo cover letter. Si no tienes tiempo para personalizarlo para esa empresa, no lo mandes.
