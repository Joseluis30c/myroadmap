Conceptos de Mensajería Asíncrona
Por qué colas, producer/consumer, exchanges, bindings, patrones Work Queue vs Pub/Sub vs Request/Reply, y diferencias RabbitMQ vs Azure Service Bus
⏱ ~5h
Una cola de mensajes permite que dos servicios se comuniquen sin estar conectados directamente ni al mismo tiempo. El producer envía un mensaje a la cola y sigue su camino. El consumer lo lee cuando puede — aunque esté en otro servidor, otro proceso, o inicie 5 minutos después. Esto resuelve tres problemas: desacoplamiento (producer y consumer no se conocen), resiliencia (si el consumer falla, el mensaje espera en la cola), y throttling (el consumer procesa a su ritmo sin importar cuántos mensajes lleguen al mismo tiempo).
💡 Para esta semana instala Redis con Docker: docker run -d --name redis -p 6379:6379 redis:7-alpine. O usa Redis Cloud (redis.io/try-free) que ofrece 30MB gratis — suficiente para aprender. Azure Cache for Redis empieza en ~$15/mes (tier C0).

Los 3 patrones de mensajería que aprenderás

Work Queue
Cola de Trabajo
Un mensaje va a exactamente UN consumer. Si hay 3 workers, cada mensaje lo procesa solo uno. Ideal para distribuir carga.
Ejemplo: procesar pedidos, resize de imágenes, envío de emails en lotes
Pub/Sub
Publicar / Suscribir
Un mensaje llega a TODOS los suscriptores. Si el pedido se creó, lo saben el servicio de email, el de inventario y el de analytics.
Ejemplo: eventos de dominio, notificaciones, sincronización entre microservicios
Request/Reply
Solicitud / Respuesta
El producer envía y espera una respuesta asíncrona. Comunicación bidireccional sin HTTP directo entre servicios.
Ejemplo: consultar precio a otro servicio, validar stock antes de confirmar

RabbitMQ — Broker local con Docker
Exchanges, queues, bindings, publicar y consumir con RabbitMQ.Client, Management UI, Dead Letter Exchange y reintentos
⏱ ~5h
RabbitMQ implementa AMQP: los mensajes no van directamente a colas, pasan por un exchange. El exchange decide a qué cola(s) enrutar el mensaje según su tipo: direct (por routing key exacta), fanout (a todas las colas vinculadas — Pub/Sub), topic (por patrón como orders.*.created), y headers. Esta separación es lo que hace a RabbitMQ flexible: cambias el routing sin tocar el código del producer ni del consumer. La Management UI en localhost:15672 permite ver mensajes, colas, bindings y el DLQ en tiempo real.
💡 Levanta RabbitMQ con Docker en 10 segundos:
docker run -d --name rabbitmq -p 5672:5672 -p 15672:15672 rabbitmq:3-management
Management UI: http://localhost:15672 — usuario/contraseña: guest / guest

Azure Service Bus — La alternativa cloud
Namespaces, queues vs topics/subscriptions, Sessions para ordering, Managed Identity, Dead Letter y Service Bus Explorer
⏱ ~5h
Azure Service Bus es el RabbitMQ gestionado de Microsoft, con garantías enterprise. La diferencia arquitectural clave: en lugar de exchanges con routing, ASB tiene Queues (un consumer) y Topics con Subscriptions (múltiples consumers filtrados). Las Sessions garantizan que todos los mensajes de un mismo SessionId los procese el mismo consumer en orden FIFO — crucial para pedidos de un cliente. El Dead Letter Queue está en el portal de Azure con interfaz visual, sin necesidad de RabbitMQ Management UI.

ServiceBusSetup.sh — Crear recursos

# Crear namespace y queue con az CLI
RG="rg-miapi-prod"
ASB_NS="asb-miapi-prod"

az servicebus namespace create \
  --resource-group "$RG" \
  --name           "$ASB_NS" \
  --location       "eastus" \
  --sku            Standard   # Basic no tiene Topics/Subscriptions

az servicebus queue create \
  --resource-group "$RG" \
  --namespace-name "$ASB_NS" \
  --name           "orders-queue" \
  --max-delivery-count 3          # 3 intentos antes del Dead Letter

# Asignar rol al Container App (Managed Identity)
az role assignment create \
  --assignee  "$APP_PRINCIPAL_ID" \
  --role      "Azure Service Bus Data Owner" \
  --scope     "$(az servicebus namespace show --name $ASB_NS --query id -o tsv)"

MassTransit — La abstracción sobre ambos brokers
Configurar MassTransit con RabbitMQ o ASB, Consumers, Sagas básicas, Outbox Pattern y tests sin broker real
⏱ ~5h
MassTransit es a la mensajería lo que Entity Framework es a las bases de datos: una abstracción que funciona sobre múltiples brokers. El mismo código de consumer funciona con RabbitMQ local en desarrollo y Azure Service Bus en producción — solo cambia la configuración en Program.cs. MassTransit agrega lo que los SDKs nativos no tienen de serie: retry policies con backoff exponencial, circuit breaker, outbox pattern para consistencia transaccional, y consumidores tipados sin boilerplate de serialización.
💡 MassTransit + Outbox Pattern = el combo definitivo. El Outbox Pattern resuelve el problema de consistencia: al crear un pedido en la BD, también guardas el mensaje en una tabla OutboxMessages dentro de la misma transacción. Un proceso en background lee la tabla y publica los mensajes. Así, si la BD falla, el mensaje no se pierde. Si el broker falla, el mensaje espera en la BD hasta que el broker vuelva.

⏱ ~5h

El caso concreto: cuando se crea un pedido, el endpoint responde 202 Accepted en <10ms, y en background el consumer procesa la orden, envía el email de confirmación y actualiza el inventario.
