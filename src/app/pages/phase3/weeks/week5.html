Cache-Aside, Write-Through
Tu API puede responder en 2ms en lugar de 400ms. Redis guarda los resultados costosos en memoria — lecturas de BD, respuestas de APIs externas, sesiones de usuario. Un endpoint sin cache que recibe 1000 req/s genera 1000 queries. Con cache: 1 query cada 5 minutos.

Las 3 estrategias de cache de esta semana

🔍
Cache-Aside
La app busca en Redis primero. Si no hay (miss), consulta la BD, guarda el resultado en Redis y retorna. El patrón más común y flexible.
Lazy loading
✏️
Write-Through
Al escribir en la BD, simultáneamente se actualiza el cache. El cache siempre tiene la versión más reciente. Cero stale data.
Always fresh
⏱️
TTL + Invalidación
Cada entrada tiene un tiempo de expiración (TTL). Al actualizar datos, se invalida la clave explícitamente. Combina ambos enfoques.
Best of both

Conceptos de Caching y Redis
Por qué cachear, tipos de cache, Redis como in-memory data store, estructuras de datos, persistencia y Redis Cloud
⏱ ~5h
Cache es una capa de almacenamiento rápido que guarda resultados costosos para reutilizarlos. Redis es el cache distribuido estándar de la industria: almacena datos en RAM (microsegundos de latencia vs milisegundos de SQL), soporta estructuras como strings, hashes, lists, sets y sorted sets, y puede distribuirse entre múltiples instancias. La diferencia con la memoria del proceso: Redis es compartido entre todas las instancias de tu API — si tienes 3 réplicas del contenedor, todas comparten el mismo cache. Sin Redis, cada réplica tendría su propia caché en memoria, causando inconsistencias.
💡 Para esta semana instala Redis con Docker: docker run -d --name redis -p 6379:6379 redis:7-alpine. O usa Redis Cloud (redis.io/try-free) que ofrece 30MB gratis — suficiente para aprender. Azure Cache for Redis empieza en ~$15/mes (tier C0).

IDistributedCache — La Abstracción de .NET
Configurar Redis con IDistributedCache, serialización con System.Text.Json, extensiones helper y caché en memoria para testing
⏱ ~5h
IDistributedCache es la abstracción oficial de .NET para cache distribuido. Inyectas la interfaz, y la implementación puede ser Redis, SQL Server o memoria — sin cambiar el código de la aplicación. La ventaja: en tests usas AddDistributedMemoryCache() (en memoria, sin Redis). En producción usas AddStackExchangeRedisCache(). Mismo código, diferente implementación. El problema: la interfaz base solo maneja byte[] — necesitas serializar/deserializar manualmente. La solución: extensiones helper que hacen esto transparente.

StackExchange.Redis — El cliente directo
ConnectionMultiplexer, operaciones batch con pipelines, Lua scripts atómicos, pub/sub y cuándo ir más allá de IDistributedCache
⏱ ~5h
StackExchange.Redis es el cliente oficial de Redis para .NET — más potente que IDistributedCache. Mientras IDistributedCache solo hace GET/SET/DELETE, StackExchange.Redis expone todas las estructuras de Redis: incrementar contadores atómicamente, operaciones en listas y sorted sets, pipelines para enviar múltiples comandos en un solo round-trip, y Lua scripts para operaciones atómicas complejas. El ConnectionMultiplexer debe ser un singleton — maneja el pool de conexiones internamente.

IDistributedCache

✗ Solo bytes — serialización manual obligatoria
✗ Solo strings (GET/SET/DELETE/EXPIRE)
✗ Sin acceso a estructuras: Hashes, Lists, Sorted Sets
✗ Sin pipelines ni operaciones batch
✗ Sin pub/sub ni Lua scripts

StackExchange.Redis

✓ Acceso completo a todas las estructuras de datos
✓ Pipelines: múltiples comandos en un round-trip
✓ INCR/DECR atómicos para contadores y rate limiting
✓ Pub/Sub para notificaciones en tiempo real
✓ Lua scripts para operaciones complejas atómicas

Estrategias de Invalidación
TTL, invalidación por evento, cache tagging, stale-while-revalidate, stampede protection y el problema del thundering herd
⏱ ~5h
La invalidación de cache es uno de los 2 problemas más difíciles de la computación. El otro es nombrar variables. Hay tres preguntas que debes responder por cada dato que cacheas: ¿Cuánto tiempo puede estar desactualizado? ¿Qué evento lo hace stale? ¿Qué pasa si 1000 requests llegan en el momento que expira? Esta última es el thundering herd problem: el cache expira, 1000 requests simultáneos ven el miss, todos van a la BD al mismo tiempo, la BD se cae. La solución es el lock distribuido o probabilistic early expiration.
⚠️ Thundering Herd: Cuando un cache popular expira, todas las requests concurrentes que ven el miss corren a la BD al mismo tiempo. Con tráfico alto, esto puede tumbar la BD. La solución más simple: usar SemaphoreSlim por clave para permitir solo una request a la BD mientras las demás esperan el resultado.
Estrategia Cuándo usar Complejidad Consistencia
TTL fijo Datos que pueden estar ligeramente desactualizados: catálogos, configuraciones, estadísticas Baja Eventual (TTL lag)
Invalidación explícita Datos críticos: perfil de usuario, precios, inventario. Al escribir, borrar la clave en Redis Media Alta
Write-Through Datos que se escriben frecuentemente y se leen aún más. Siempre consistente, nunca stale Media Inmediata
Cache Tags Invalidar grupos de claves relacionadas. Ej: "todos los productos de la categoría X" Alta Alta
Stale-While-Revalidate Datos tolerantes a staleness: dashboards, feeds. Retorna el dato viejo inmediatamente mientras actualiza en background Media Eventual (ms)

Endpoint Completo con Cache + Azure Cache for Redis
Cache layer sobre repositorio, Output Cache en .NET 8, Azure Cache for Redis, métricas de hit/miss y el entregable final
⏱ ~5h
.NET 8 introduce Output Caching — la forma más simple de cachear endpoints enteros. Con [OutputCache(Duration = 300)] en el endpoint, el framework serializa la respuesta completa y la guarda en el cache automáticamente. Zero boilerplate. Para lógica más compleja (invalidación por evento, cache by user, queries dinámicas), se usa IDistributedCache o StackExchange.Redis directamente.
