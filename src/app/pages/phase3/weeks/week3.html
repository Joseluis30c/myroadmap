Conceptos Docker — Images, Containers, Registry
Image vs Container, Docker Hub, comandos esenciales, capas, volúmenes, redes y el daemon de Docker
⏱ ~5h
Docker empaqueta tu aplicación con todo lo que necesita para correr: runtime, librerías, configuración del sistema operativo. Una image es el paquete inmutable — como una clase. Un container es una instancia corriendo de esa imagen — como un objeto. El registry (Docker Hub, Azure Container Registry) es donde guardas y distribuyes las imágenes. La diferencia fundamental con una VM: los contenedores comparten el kernel del host — arrancan en milisegundos y usan MBs de RAM en lugar de GBs.
💡 Instala Docker Desktop en tu máquina: incluye Docker Engine, Docker CLI, Docker Compose y un dashboard visual. En Windows requiere WSL2. Gratuito para uso personal y educativo: docker.com/products/docker-desktop

# ─── CONTENEDORES ──────────────────────────────────────────────
docker run -d \
  --name miapi \
  -p 8080:80 \                     # host:container port mapping
  -e ASPNETCORE_ENVIRONMENT=Development \
  -e ConnectionStrings__Default="..." \
  --restart unless-stopped \       # reiniciar si el contenedor se cae
  miapi:1.0

docker ps                          # contenedores corriendo
docker ps -a                       # todos (incluye detenidos)
docker logs miapi                  # ver logs
docker logs -f miapi               # seguir logs en tiempo real
docker exec -it miapi /bin/bash    # shell interactivo dentro del contenedor
docker stop miapi                  # detener (graceful)
docker rm -f miapi                 # forzar eliminación
docker stats                       # CPU/RAM de contenedores en tiempo real
docker inspect miapi               # metadatos completos (JSON)

# ─── VOLÚMENES (persistencia de datos) ─────────────────────────

# En la misma red, los contenedores se ven por nombre:
# connection string: Server=sqlserver;  (nombre del contenedor, no localhost)

# ─── LIMPIEZA ──────────────────────────────────────────────────

Dockerfile para ASP.NET Core — Multi-stage Build
Instrucciones, orden de capas para caché óptimo, imagen de 100MB en vez de 700MB, .dockerignore y non-root user
⏱ ~5h
El Dockerfile más importante para .NET usa multi-stage build: el primer stage usa la imagen SDK completa (~700MB) para compilar y publicar. El segundo stage copia solo el binario publicado a la imagen runtime mínima (~100MB). El SDK no va a producción — solo el resultado. El orden de las instrucciones es crítico: COPY *.csproj y RUN dotnet restore primero, antes de COPY . . — así Docker cachea las dependencias y el restore solo corre cuando el .csproj cambia, no con cada cambio de código.

Dockerfile — Anatomía de capas y caché | multi-stage · .NET 8

1 FROM sdk:8.0 AS build ~730MB
2 WORKDIR /src 0B
3 COPY *.csproj . → (solo el .csproj) CACHE KEY ~3KB
4 RUN dotnet restore → descarga NuGet packages CACHED si .csproj no cambió
5 COPY . . → (todo el código fuente) código
6 RUN dotnet publish -c Release -o /app/publish ~15MB
7 FROM aspnet:8.0 AS final → nueva imagen limpia ~100MB
8 COPY --from=build /app/publish . → solo el binario ~15MB
9 ENTRYPOINT ["dotnet", "Api.dll"] 0B

Stage build: ~730MB (desechado) | Imagen final: ~115MB (lo que se deploya)

💻 Código — Dockerfile production-ready para ASP.NET Core

Multi-stage build + non-root user + health check + .dockerignore

Dockerfile

# ─── STAGE 1: BUILD ─────────────────────────────────────────────
FROM mcr.microsoft.com/dotnet/sdk:8.0 AS build
WORKDIR /src

# Copiar SOLO los .csproj primero → cachear restore por separado
COPY ["src/Api/Api.csproj",          "src/Api/"]
COPY ["src/Application/Application.csproj", "src/Application/"]
COPY ["src/Infrastructure/Infrastructure.csproj", "src/Infrastructure/"]
COPY ["src/Domain/Domain.csproj",      "src/Domain/"]

# Restore: solo corre si algún .csproj cambió
RUN dotnet restore "src/Api/Api.csproj"

# Ahora copiar el código fuente (invalida caché solo si el código cambia)
COPY . .

RUN dotnet publish "src/Api/Api.csproj" \
    -c Release \
    --no-restore \
    -o /app/publish \
    /p:UseAppHost=false

# ─── STAGE 2: FINAL (runtime solamente) ─────────────────────────
FROM mcr.microsoft.com/dotnet/aspnet:8.0 AS final
WORKDIR /app

# Non-root user: nunca correr como root en producción
RUN addgroup --system appgroup && adduser --system --ingroup appgroup appuser

# Copiar solo el binario publicado del stage anterior
COPY --from=build /app/publish .

# Cambiar propietario de los archivos
RUN chown -R appuser:appgroup /app
USER appuser

# Puerto que expone la app (documentación, no abre el puerto)
EXPOSE 8080
EXPOSE 8081  # HTTPS

# Health check: Docker verificará /health cada 30s
HEALTHCHECK --interval=30s --timeout=10s --start-period=30s --retries=3 \
    CMD wget -qO- http://localhost:8080/health || exit 1

ENTRYPOINT ["dotnet", "Api.dll"]
.dockerignore

# Excluir lo que no necesita Docker para el build
**/.git
**/.vs
**/bin
**/obj
**/*.user
**/node_modules
**/.env                   # NUNCA incluir .env con secrets
**/docker-compose*.yml    # no necesario en la imagen
**/README.md
**/tests                  # tests no van en imagen de producción

Variables de Entorno y Configuración
ENV vs ARG, .env files, Docker Secrets, IConfiguration en .NET y cómo connection strings fluyen de Compose a la app
⏱ ~5h
Las variables de entorno son la única forma correcta de configurar contenedores. En .NET, IConfiguration lee automáticamente variables de entorno con el prefijo de la jerarquía usando doble guion bajo: ConnectionStrings__DefaultConnection equivale a appsettings.json → ConnectionStrings → DefaultConnection. Nunca hardcodees secrets en el Dockerfile ni los comittees en el repositorio. El archivo .env es para desarrollo local — en producción (Azure, Kubernetes) se inyectan como variables de entorno del contenedor o como Docker Secrets.
⚠️ El archivo .env NUNCA va al repositorio. Agrega .env al .gitignore y al .dockerignore ahora mismo. Si alguien ya committeó secrets, revoca las credenciales y considera el historial de git comprometido.

Variables de entorno para tu API .NET en Docker

Variable Ejemplo Obligatoria Descripción
ASPNETCORE_ENVIRONMENT Production Sí Determina qué appsettings.{{'{'}}env{{'}'}}.json se carga
ASPNETCORE_HTTP_PORTS 8080 Sí Puerto HTTP interno del contenedor (.NET 8+)
ConnectionStrings__DefaultConnection Server=db;Database=...;User Id=sa; Sí __ = separador jerárquico que lee IConfiguration
Jwt__Secret super-secret-key-32-chars-min Sí Clave de firma JWT — mínimo 32 caracteres
Storage__ConnectionString DefaultEndpointsProtocol=https;... Opcional Azure Blob Storage connection string
DOTNET_RUNNING_IN_CONTAINER true Auto Seteada automáticamente por las imágenes oficiales de .NET

Docker Compose — API + SQL Server + Redis
docker-compose.yml, services, redes, volúmenes, depends_on con healthcheck, override para dev y prod
⏱ ~5h
Docker Compose orquesta múltiples contenedores como un ambiente completo con un solo comando. En lugar de levantar manualmente SQL Server, Redis y tu API en tres terminales, docker compose up -d lo hace todo: crea la red interna, levanta los servicios en orden correcto (respetando depends_on con healthchecks), monta los volúmenes para persistir datos. El nombre del servicio es el hostname que usan los otros servicios — tu connection string apunta a Server=db, no a localhost.

docker-compose.yml — Arquitectura de servicios | miapi_default network (bridge)

⚙️ api
image: miapi:latest
ports: 8080:8080
depends: db, redis
env_file: .env
↔
🗄️db
image: mssql:2022
ports: 1433:1433
volume: sqldata:/data
health: sqlcmd SELECT 1
↔
⚡redis
image: redis:7-alpine
ports: 6379:6379
volume: redisdata:/data
health: redis-cli ping

Todos los servicios en la misma red → se ven por nombre de servicio | Server=db; no Server=localhost

💻 Código — Docker Compose completo con healthchecks y override

API + SQL Server + Redis con depends_on correcto y override para dev

docker-compose.yml

name: miapi

services:

  # ── SQL Server ─────────────────────────────────────────────────
  db:
    image: mcr.microsoft.com/mssql/server:2022-latest
    environment:
      SA_PASSWORD:        ${{SA_PASSWORD}}
      ACCEPT_EULA:        Y
      MSSQL_PID:          Developer
    ports:
      - "1433:1433"              # expuesto solo para SSMS en dev
    volumes:
      - sqldata:/var/opt/mssql  # datos persisten entre compose up/down
    healthcheck:
      test: ["CMD", "/opt/mssql-tools18/bin/sqlcmd",
             "-S", "localhost", "-U", "sa",
             "-P", "${{SA_PASSWORD}}", "-Q", "SELECT 1", "-No"]
      interval:     15s
      timeout:      10s
      retries:      5
      start_period: 30s        # SQL Server tarda ~20s en arrancar

  # ── Redis ──────────────────────────────────────────────────────
  redis:
    image: redis:7-alpine
    volumes:
      - redisdata:/data
    healthcheck:
      test:      ["CMD", "redis-cli", "ping"]
      interval: 10s
      timeout:  5s
      retries:  3

  # ── API ────────────────────────────────────────────────────────
  api:
    build:
      context:    .
      dockerfile: Dockerfile
    ports:
      - "8080:8080"
    env_file:
      - .env                       # .env en .gitignore siempre
    environment:
      # Connection string apunta al NOMBRE DEL SERVICIO, no a localhost
      ConnectionStrings__DefaultConnection:
        "Server=db;Database=${{DB_NAME}};User Id=sa;Password=${{SA_PASSWORD}};TrustServerCertificate=True"
      Redis__ConnectionString:
        "redis:6379"              # nombre del servicio redis
    depends_on:
      db:
        condition: service_healthy  # espera hasta que db pase el healthcheck
      redis:
        condition: service_healthy
    restart: on-failure

# Volúmenes nombrados: persisten entre compose down/up
volumes:
  sqldata:
  redisdata:

💻 Código — GitHub Actions: build imagen Docker + push a ACR

Actualizar el pipeline de la Semana 12 para deployar contenedores en vez de binarios

.github/workflows/docker-deploy.yml (fragmento)
                        
# Agregar estos steps al job de deploy de la Semana 12
jobs:
  build-and-push:
    runs-on: ubuntu-latest
    permissions:
      id-token:  write
      contents: read
    steps:
      - uses: actions/checkout@v4

      # Login a Azure con OIDC (igual que la Semana 12)
      - uses: azure/login@v2
        with:
          client-id:       ${{ secrets.AZURE_CLIENT_ID }}
          tenant-id:       ${{ secrets.AZURE_TENANT_ID }}
          subscription-id: ${{ secrets.AZURE_SUBSCRIPTION_ID }}

      # Login al Azure Container Registry
      - name: Login to ACR
        run: az acr login --name ${{ secrets.ACR_NAME }}

      # Build y push con tag = SHA del commit
      - name: Build + Push Docker Image
        run: |
          IMAGE="${{ secrets.ACR_NAME }}.azurecr.io/miapi"
          TAG="${{ github.sha }}"
          docker build -t "$IMAGE:$TAG" -t "$IMAGE:latest" .
          docker push "$IMAGE:$TAG"
          docker push "$IMAGE:latest"
          echo "IMAGE_TAG=$TAG" >> $GITHUB_OUTPUT

      # Deploy la nueva imagen a Container Apps (staging)
      - name: Update Container App
        run: |
          az containerapp update \
            --name      "${{ secrets.CONTAINER_APP_NAME_STG }}" \
            --resource-group "${{ secrets.AZURE_RESOURCE_GROUP }}" \
            --image    "${{ secrets.ACR_NAME }}.azurecr.io/miapi:${{ github.sha }}"
