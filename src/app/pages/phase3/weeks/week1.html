Azure Portal, Subscriptions, CLI y Conceptos Core
Subscriptions, Resource Groups, Regions, az CLI, nomenclatura de recursos y el modelo de precios
⏱ ~5h
Azure organiza todo en una jerarquía: Management Group → Subscription → Resource Group → Resource. Tu punto de entrada es la Subscription (facturación y límites), y dentro de ella creas Resource Groups como contenedores lógicos para todos los recursos de un proyecto. Todo lo que crees en este grupo se puede eliminar en bloque con az group delete --name rg-miapi-dev — perfecto para limpiar entornos de prueba. La región determina la latencia física y debe ser la misma para todos los recursos que se comunican entre sí.
💡Cuenta Gratuita: Azure ofrece $200 de crédito por 30 días + 12 meses de servicios gratuitos (incluyendo App Service B1, Azure SQL 250GB, Blob Storage 5GB). Crea tu cuenta en portal.azure.com — todo lo de esta semana cabe en el free tier.

Azure App Service — Deploy de tu API .NET
Planes de servicio, tiers gratuitos vs pagos, deployment slots, App Settings como variables de entorno y zip deploy
⏱ ~5h
App Service es PaaS — tú subes el código, Azure maneja el servidor. Sin parches de SO, sin administración de IIS, sin configuración de TLS. Eliges un App Service Plan (las VMs subyacentes) y creas tantas apps como quieras en ese plan. Para desarrollo usa B1 (~$13/mes o gratis con la cuenta free). Para producción, S1 agrega deployment slots (blue/green deploy) y auto-scaling. Las App Settings son variables de entorno que sobreescriben el appsettings.json — la forma correcta de manejar connection strings en la nube.
NUNCA pongas connection strings en el código fuente. Usa App Settings en Azure (o Azure Key Vault para producción). El App Service inyecta esas variables como environment variables en tiempo de ejecución, que IConfiguration lee automáticamente en .NET.
Tier RAM / CPU Precio Características clave ¿Para qué?
Free F1 1GB / shared Gratis 60 min CPU/día, sin custom domain, sin SSL Solo aprender
Free B1 1.75GB / 1 core ~$13/mes Custom domain, SSL, sin auto-scale Dev/esta semana
Standard S1 1.75GB / 1 core ~$73/mes Deployment slots, auto-scale, VNet Staging/QA
Premium P1v3 8GB / 2 cores ~$138/mes Mejor CPU, private endpoints, zone redundancy Producción

💻 Código — Crear App Service + deploy con az CLI

De cero a API desplegada en Azure en 10 comandos

deploy_appservice.sh
# ─── Crear App Service Plan (las VMs) ───
az appservice plan create \
  --name      "plan-miapi-dev" \
  --resource-group "$RESOURCE_GROUP" \
  --sku       "B1" \
  --is-linux  \
  --location  "$LOCATION"

# ─── Crear la Web App ───
az webapp create \
  --name         "$APP_NAME" \
  --resource-group "$RESOURCE_GROUP" \
  --plan         "plan-miapi-dev" \
  --runtime      "DOTNETCORE:8.0"

# Verificar URL
az webapp show \
  --name "$APP_NAME" \
  --resource-group "$RESOURCE_GROUP" \
  --query defaultHostName \
  --output tsv
# → miapi-XXXXXXX.azurewebsites.net

# ─── Configurar App Settings (variables de entorno) ───
az webapp config appsettings set \
  --name "$APP_NAME" \
  --resource-group "$RESOURCE_GROUP" \
  --settings \
    "ASPNETCORE_ENVIRONMENT=Production" \
    "ConnectionStrings__DefaultConnection=Server=...;Database=...;" \
    "Jwt__Secret=@Microsoft.KeyVault(VaultName=kv-miapi;SecretName=jwt-secret)"

# ─── Deploy desde carpeta publicada (zip deploy) ───
dotnet publish -c Release -o ./publish
cd publish
zip -r ../deploy.zip .
cd ..

az webapp deployment source config-zip \
  --name "$APP_NAME" \
  --resource-group "$RESOURCE_GROUP" \
  --src "deploy.zip"

# ─── Ver logs en tiempo real ───
az webapp log tail \
  --name "$APP_NAME" \
  --resource-group "$RESOURCE_GROUP"

Azure SQL Database — Tu BD en la nube
Logical server vs database, DTU vs vCore, firewall rules, connection strings para EF Core, backups automáticos y migration deploy
⏱ ~5h
Azure SQL Database es SQL Server completamente administrado. Todo tu código T-SQL, tus índices, tus stored procedures, EF Core — todo funciona sin cambiar una línea. La arquitectura tiene dos niveles: el Logical Server (punto de conexión, firewall, logins) y las Databases (los datos reales, el tier de performance). Para desarrollo usa Basic (5 DTU) a ~$5/mes. Para producción, el modelo Serverless es ideal: escala CPU a cero cuando no hay tráfico y se "wake up" automáticamente.
💡 Firewall rules son obligatorias: Por defecto, Azure SQL no acepta conexiones de nadie. Debes agregar el IP de tu App Service (o habilitar "Allow Azure services") y el IP de tu máquina de desarrollo. Jamás uses 0.0.0.0 – 255.255.255.255 — eso expone tu BD a internet completo.

Crear la BD, configurar EF Core y ejecutar migrations en Azure

azure_sql_setup.sh
# ─── Crear SQL Server Lógico ───
az sql server create \
  --name           "$SQL_SERVER" \
  --resource-group "$RESOURCE_GROUP" \
  --location       "$LOCATION" \
  --admin-user     "sqladmin" \
  --admin-password "P@ssw0rd!2024"

# ─── Crear la Database (Basic = 5 DTU = ~$5/mes) ───
az sql db create \
  --name           "$SQL_DB" \
  --resource-group "$RESOURCE_GROUP" \
  --server         "$SQL_SERVER" \
  --edition        "Basic"

# ─── Firewall: permitir tu IP actual ───
MY_IP=$(curl -s https://api.ipify.org)
az sql server firewall-rule create \
  --server         "$SQL_SERVER" \
  --resource-group "$RESOURCE_GROUP" \
  --name           "AllowMyDev" \
  --start-ip-address "$MY_IP" \
  --end-ip-address   "$MY_IP"

# Permitir que App Service se conecte (IPs de Azure)
az sql server firewall-rule create \
  --server         "$SQL_SERVER" \
  --resource-group "$RESOURCE_GROUP" \
  --name           "AllowAzureServices" \
  --start-ip-address "0.0.0.0" \
  --end-ip-address   "0.0.0.0"

# ─── Obtener connection string ───
az sql db show-connection-string \
  --server "$SQL_SERVER" \
  --name   "$SQL_DB" \
  --client ado.net

Azure Blob Storage + Conceptos de Red
Storage accounts, containers, blobs, SAS tokens, .NET SDK, VNet, NSG, Service Endpoints y configuración de firewall segura
⏱ ~5h
Blob Storage es el almacenamiento de objetos de Azure — archivos, imágenes, backups, cualquier binario. La jerarquía es: Storage Account → Container (como una carpeta) → Blob (el archivo). El SDK de Azure.Storage.Blobs para .NET hace el upload/download en pocas líneas. Para networking, el concepto clave es que tu App Service y Azure SQL deben comunicarse por la red interna de Azure usando Service Endpoints o Private Endpoints, no por internet público.

💻 Código — Deploy completo + Application Insights

El script final que deja tu API corriendo en Azure

#!/bin/bash
# ─── Script de deploy completo — Semana 11 ───

set -e  # detener si hay error

RESOURCE_GROUP="rg-miapi-dev"
APP_NAME="app-miapi-dev"
SQL_SERVER="sql-miapi-dev"
SQL_DB="db-miapi"
STORAGE_ACC="stmiapistg"

# 1. Obtener connection strings
SQL_CONN=$(az sql db show-connection-string \
  --server "$SQL_SERVER" --name "$SQL_DB" \
  --client ado.net --auth-type SqlPassword \
  --output tsv)

STORAGE_CONN=$(az storage account show-connection-string \
  --name "$STORAGE_ACC" \
  --resource-group "$RESOURCE_GROUP" \
  --output tsv)

APPINS_KEY=$(az monitor app-insights component show \
  --app "appins-miapi" \
  --resource-group "$RESOURCE_GROUP" \
  --query instrumentationKey --output tsv)

# 2. Configurar App Settings
az webapp config appsettings set \
  --name "$APP_NAME" \
  --resource-group "$RESOURCE_GROUP" \
  --settings \
    "ConnectionStrings__DefaultConnection=$SQL_CONN" \
    "Storage__ConnectionString=$STORAGE_CONN" \
    "ApplicationInsights__InstrumentationKey=$APPINS_KEY" \
    "ASPNETCORE_ENVIRONMENT=Production"

# 3. Publicar y deployar
dotnet publish ./src/Api -c Release -o ./dist

pushd dist
zip -r ../release.zip . -x "*.pdb"
popd

az webapp deployment source config-zip \
  --name "$APP_NAME" \
  --resource-group "$RESOURCE_GROUP" \
  --src release.zip

# 4. Verificar health endpoint
URL=$(az webapp show \
  --name "$APP_NAME" \
  --resource-group "$RESOURCE_GROUP" \
  --query defaultHostName --output tsv)

echo "✓ Deploy completo: https://$URL"
curl -s "https://$URL/health" | jq
