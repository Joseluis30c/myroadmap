Azure Container Registry — Tu registro privado
Crear ACR, SKUs, login con OIDC, push/pull de imágenes, geo-replication y políticas de retención
⏱ ~5h
Azure Container Registry (ACR) es el Docker Hub privado de Azure. En lugar de guardar tus imágenes en Docker Hub público, las guardas en un registro privado dentro de tu subscripción de Azure. Solo tus servicios de Azure (Container Apps, AKS, App Service) tienen acceso. El SKU Basic (~$5/mes) es suficiente para desarrollo. Standard agrega geo-replication y mayor throughput para producción. La integración con Managed Identity elimina la necesidad de guardar credenciales del registro en ningún lugar.
💡 Usa az acr build en lugar de docker build + docker push. Este comando construye la imagen directamente en Azure usando los recursos del ACR — sin descargar el SDK de .NET en el runner de GitHub Actions. El build es más rápido y no consume los minutos del runner para compilar.

💻 Código — Crear ACR y hacer push de la primera imagen

De cero a imagen en el registro privado con az CLI

acr_setup.sh

# Variables (ajusta a tu proyecto)
RESOURCE_GROUP="rg-miapi-prod"
LOCATION="eastus"
ACR_NAME="acrmiapi"          # único globalmente, solo letras/números
APP_NAME="miapi"
IMAGE_TAG=$(git rev-parse --short HEAD)

# ─── 1. Crear Azure Container Registry ───────────────────────────
az acr create \
  --resource-group "$RESOURCE_GROUP" \
  --name           "$ACR_NAME" \
  --sku            "Basic" \
  --admin-enabled  "false"    # no usar admin creds — usar Managed Identity

# ─── 2. Build remoto (se compila en Azure, no en tu máquina) ──────
# No necesitas Docker instalado para este comando
az acr build \
  --registry   "$ACR_NAME" \
  --image      "$APP_NAME:$IMAGE_TAG" \
  --image      "$APP_NAME:latest" \
  --file       "Dockerfile" \
  .

# Verificar que la imagen se subió
az acr repository show-tags \
  --name       "$ACR_NAME" \
  --repository "$APP_NAME" \
  --output     table

# ─── 3. Política de retención: borrar imágenes con más de 30 días ─
az acr config retention update \
  --registry "$ACR_NAME" \
  --status   enabled \
  --days     30 \
  --type     UntaggedManifests

# ─── 4. Login para pull manual (dev) ─────────────────────────────
az acr login --name "$ACR_NAME"
docker pull "$ACR_NAME.azurecr.io/$APP_NAME:latest"

Azure Container Apps — Deploy del contenedor
Crear el Container App, asignar Managed Identity, pull desde ACR, scaling rules, revisiones y blue/green deploy
⏱ ~5h
Azure Container Apps (ACA) es la forma más sencilla de correr contenedores en Azure sin administrar Kubernetes. Cada deploy crea una nueva revisión — una versión inmutable de tu contenedor. Puedes dividir el tráfico: "80% a la revisión actual, 20% a la nueva" para blue/green deploy sin downtime. Escala a cero automáticamente cuando no hay tráfico (ideal para dev/staging — pagas cero cuando nadie lo usa). La Managed Identity le permite al Container App acceder a ACR y Key Vault sin guardar ninguna credencial.

Pasos para el primer deploy

1 Crear Container Apps Environment
El environment es el namespace compartido para tus Container Apps — configura la red, el Log Analytics workspace y el dominio base. Un environment por entorno (staging, production).		 🌐
Environment
2 Crear Container App con Managed Identity
Crear la app apuntando a la imagen en ACR. Asignar AcrPull role para que pueda descargar imágenes sin contraseñas. La identity también se usa para Key Vault.		 🔑
Identity
3 Configurar Ingress y puerto
Habilitar ingress externo en el puerto que expone tu contenedor (8080 en .NET 8). ACA genera automáticamente una URL HTTPS con certificado TLS válido: https://miapi.azurecontainerapps.io		 🌍
HTTPS
4 Configurar variables de entorno y secrets
Las variables de entorno se configuran en el Container App igual que en Docker Compose. Los valores sensibles (connection strings, JWT secret) se referencian desde Key Vault.		 ⚙️
Config
5 Configurar scaling rules
Min replicas: 0 (escala a cero para dev). Max replicas: 3–10 para producción. Regla HTTP: 1 réplica por cada 100 requests concurrentes. Las réplicas arrancan en ~3s.		 📈
Scaling

💻 Código — Crear Container App completo con az CLI

Environment, Managed Identity, ingress, secrets y scaling en un script

container_app_deploy.sh

# ─── Variables ────────────────────────────────────────────────────
RG="rg-miapi-prod"
LOCATION="eastus"
ENV_NAME="cae-miapi-prod"
APP_NAME="miapi"
ACR_NAME="acrmiapi"
IMAGE_TAG="latest"
LOG_WS="log-miapi-prod"

# ─── 1. Log Analytics Workspace (para los logs del contenedor) ───
az monitor log-analytics workspace create \
  --resource-group "$RG" \
  --workspace-name "$LOG_WS" \
  --location       "$LOCATION"

LOG_WS_ID=$(az monitor log-analytics workspace show \
  --resource-group "$RG" \
  --workspace-name "$LOG_WS" \
  --query customerId --output tsv)

LOG_WS_KEY=$(az monitor log-analytics workspace get-shared-keys \
  --resource-group "$RG" \
  --workspace-name "$LOG_WS" \
  --query primarySharedKey --output tsv)

# ─── 2. Container Apps Environment ────────────────────────────────
az containerapp env create \
  --name                      "$ENV_NAME" \
  --resource-group            "$RG" \
  --location                  "$LOCATION" \
  --logs-workspace-id         "$LOG_WS_ID" \
  --logs-workspace-key        "$LOG_WS_KEY"

# ─── 3. Container App con Managed Identity ────────────────────────
az containerapp create \
  --name                     "$APP_NAME" \
  --resource-group           "$RG" \
  --environment              "$ENV_NAME" \
  --image                    "$ACR_NAME.azurecr.io/$APP_NAME:$IMAGE_TAG" \
  --registry-server          "$ACR_NAME.azurecr.io" \
  --registry-identity        system                  \
  --target-port              8080                    \
  --ingress                  external                \
  --min-replicas             0                       \
  --max-replicas             5                       \
  --cpu                      0.5                     \
  --memory                   "1Gi"                   \
  --env-vars \
    "ASPNETCORE_ENVIRONMENT=Production" \
    "ASPNETCORE_HTTP_PORTS=8080"

# ─── 4. Asignar rol AcrPull a la Managed Identity ─────────────────
APP_PRINCIPAL_ID=$(az containerapp show \
  --name           "$APP_NAME" \
  --resource-group "$RG" \
  --query          identity.principalId --output tsv)

ACR_ID=$(az acr show --name "$ACR_NAME" --query id --output tsv)

az role assignment create \
  --assignee   "$APP_PRINCIPAL_ID" \
  --role       "AcrPull" \
  --scope      "$ACR_ID"

# ─── 5. Ver la URL pública ─────────────────────────────────────────
az containerapp show \
  --name           "$APP_NAME" \
  --resource-group "$RG" \
  --query          properties.configuration.ingress.fqdn \
  --output         tsv
# → miapi.kindbeach-a1b2c3d4.eastus.azurecontainerapps.io

Variables Seguras — Key Vault + Managed Identity
Azure Key Vault para secrets, Managed Identity vs Service Principal, integración con IConfiguration y rotación de secrets
⏱ ~5h
Azure Key Vault es la solución definitiva para secretos en producción. Tu API accede a Key Vault usando Managed Identity — Azure le asigna una identidad a tu Container App automáticamente. No hay contraseñas, no hay client secrets, no hay nada que rotar manualmente. La integración con IConfiguration de .NET es transparente: instalas el paquete Azure.Extensions.AspNetCore.Configuration.Secrets y los secretos aparecen en builder.Configuration["NombreDelSecreto"] igual que si fueran variables de entorno.

❌ Sin Key Vault (no hacer esto)

✗ Connection string en App Settings como texto plano — visible para cualquiera con acceso al portal
✗ JWT secret en variable de entorno — aparece en los logs de deploy y en el historial de az CLI
✗ Credenciales en el código fuente o en el Dockerfile — en el historial de git para siempre
✗ Rotar una contraseña requiere redeploy manual de todos los servicios que la usan

✅ Con Key Vault (la forma correcta)

✓ Secrets en Key Vault — acceso controlado por RBAC, auditoría de cada acceso, versiones históricas
✓ Container App accede via Managed Identity — cero credenciales almacenadas en ningún archivo
✓ IConfiguration los lee como si fueran variables de entorno — cero cambios en el código
✓ Rotar un secret en Key Vault actualiza automáticamente todos los servicios que lo usan

💻 Código — Key Vault + DefaultAzureCredential en ASP.NET Core

Crear el vault, agregar secrets y leerlos en la API sin contraseñas

keyvault_setup.sh

# ─── 1. Crear Key Vault ────────────────────────────────────────────
KV_NAME="kv-miapi-prod"

az keyvault create \
  --name           "$KV_NAME" \
  --resource-group "$RG" \
  --location       "$LOCATION" \
  --sku            standard \
  --enable-rbac-authorization true   # RBAC en lugar de Access Policies

# ─── 2. Agregar secrets ───────────────────────────────────────────
az keyvault secret set \
  --vault-name "$KV_NAME" \
  --name       "DefaultConnection" \
  --value      "Server=sql-miapi-prod.database.windows.net;Database=..."

az keyvault secret set \
  --vault-name "$KV_NAME" \
  --name       "JwtSecret" \
  --value      "$(openssl rand -base64 48)"  # generar secreto random

az keyvault secret set \
  --vault-name "$KV_NAME" \
  --name       "StorageConnectionString" \
  --value      "DefaultEndpointsProtocol=https;..."

# ─── 3. Dar acceso a la Managed Identity del Container App ────────
KV_ID=$(az keyvault show --name "$KV_NAME" --query id --output tsv)

az role assignment create \
  --assignee  "$APP_PRINCIPAL_ID" \
  --role      "Key Vault Secrets User" \
  --scope     "$KV_ID"

Logs en Producción — Log Analytics + App Insights
KQL queries, structured logging con Serilog, alertas automáticas, dashboards y diagnóstico de errores en producción
⏱ ~5h
En producción no tienes acceso a la consola — los logs son tu única visibilidad. Azure Container Apps envía los logs del contenedor automáticamente a Log Analytics. Desde ahí, KQL (Kusto Query Language) permite consultar millones de líneas de log en segundos: "todos los errores 500 de las últimas 2 horas", "latencia promedio del endpoint /api/orders por minuto", "qué requests fallaron justo antes del pico de CPU". Application Insights agrega tracing distribuido — puedes ver la cadena completa de una request: API → SQL → Redis → respuesta.
💡 Serilog con structured logging transforma tus logs. En lugar de _logger.LogInformation($"Order {{'{'}}id{{'}'}} created") (texto plano imposible de filtrar), usas _logger.LogInformation("Order {{'{'}}OrderId{{'}'}} created by {{'{'}}UserId{{'}'}}", id, userId) — KQL puede filtrar por OrderId o UserId como columnas independientes.

Queries KQL esenciales para diagnóstico en producción

Qué quieres ver Query KQL Cuándo usarlo
Todos los errores recientes ContainerAppConsoleLogs | where Log contains "ERROR" | order by TimeGenerated desc Primera acción cuando reportan un bug
Latencia por endpoint requests | summarize avg(duration) by name | order by avg_duration desc Identificar endpoints lentos
Error rate por hora requests | summarize errors=countif(success==false), total=count() by bin(timestamp,1h) Ver tendencias de errores
Exceptions con stack trace exceptions | where timestamp > ago(24h) | project timestamp, type, outerMessage, details Entender qué excepción se lanzó
Requests lentas (>2s) requests | where duration > 2000 | project timestamp, name, duration, resultCode Encontrar candidatos a optimizar
Logs de un contenedor específico ContainerAppConsoleLogs | where ContainerName == "miapi" | order by TimeGenerated desc Debug de un pod específico

alert_setup.sh — Alerta cuando error rate > 5%
# Crear Action Group (a quién notificar)
az monitor action-group create \
  --resource-group "$RG" \
  --name           "ag-miapi-oncall" \
  --short-name     "miapi" \
  --email-receivers name="dev" email="dev@miempresa.com"

# Crear alerta: error rate > 5% en los últimos 5 minutos
az monitor scheduled-query create \
  --resource-group     "$RG" \
  --name               "alert-error-rate" \
  --scopes             "$APPINSIGHTS_ID" \
  --condition-query    "requests | where timestamp > ago(5m) | summarize errorRate = 100.0*countif(success==false)/count() | where errorRate > 5" \
  --condition-threshold 0 \
  --condition-operator  "GreaterThan" \
  --evaluation-frequency "5m" \
  --window-size        "5m" \
  --severity           2 \
  --action             "ag-miapi-oncall" \
  --description        "Error rate above 5% in last 5 minutes"

💻 Código — Pipeline completo GitHub Actions → ACR → Container Apps

El workflow final de la Semana 14: build en ACR + deploy + rollback automático

.github/workflows/docker-azure.yml

name: 🐳 Docker → Azure

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

permissions:
  id-token:  write
  contents: read

env:
  ACR_NAME:  acrmiapi
  IMAGE:     miapi

jobs:

  # ── CI: Tests ─────────────────────────────────────────────────────
  test:
    name:     ✅ Test
    runs-on:  ubuntu-latest
    services:
      sqlserver:
        image:  mcr.microsoft.com/mssql/server:2022-latest
        env:
          SA_PASSWORD:  TestPass!2024
          ACCEPT_EULA:  Y
        ports:
          - 1433:1433
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-dotnet@v4
        with:
          dotnet-version: '8.0.x'
      - run:  dotnet test --configuration Release
        env:
          TEST_DB: "Server=localhost;Database=TestDb;User Id=sa;Password=TestPass!2024;TrustServerCertificate=True"

  # ── Build imagen en ACR (sin Docker instalado en el runner) ───────
  build:
    name:    🔨 Build → ACR
    needs:   test
    runs-on: ubuntu-latest
    if:      github.ref == 'refs/heads/main'
    outputs:
      image-tag: ${{ steps.build.outputs.tag }}
    steps:
      - uses: actions/checkout@v4
      - uses: azure/login@v2
        with:
          client-id:       ${{ secrets.AZURE_CLIENT_ID }}
          tenant-id:       ${{ secrets.AZURE_TENANT_ID }}
          subscription-id: ${{ secrets.AZURE_SUBSCRIPTION_ID }}
      - name: Build en ACR
        id:   build
        run: |
          TAG="${{ github.sha }}"
          az acr build \
            --registry "${{ env.ACR_NAME }}" \
            --image    "${{ env.IMAGE }}:$TAG" \
            --image    "${{ env.IMAGE }}:latest" \
            --file     Dockerfile .
          echo "tag=$TAG" >> $GITHUB_OUTPUT

  # ── Deploy Staging (automático) ───────────────────────────────────
  staging:
    name:        🟡 Staging
    needs:       build
    runs-on:     ubuntu-latest
    environment: staging
    steps:
      - uses: azure/login@v2
        with:
          client-id:       ${{ secrets.AZURE_CLIENT_ID }}
          tenant-id:       ${{ secrets.AZURE_TENANT_ID }}
          subscription-id: ${{ secrets.AZURE_SUBSCRIPTION_ID }}
      - run: |
          az containerapp update \
            --name "${{ secrets.ACA_NAME_STG }}" \
            --resource-group "${{ secrets.RG }}" \
            --image "${{ env.ACR_NAME }}.azurecr.io/${{ env.IMAGE }}:${{ needs.build.outputs.image-tag }}"
      - name: 🩺 Smoke Test
        run: |
          sleep 15  # esperar que el contenedor arranque
          STATUS=$(curl -sf -o /dev/null -w "%{{http_code}}" \
            "https://${{ secrets.ACA_URL_STG }}/health")
          [ "$STATUS" = "200" ] || (echo "❌ Health check failed: $STATUS" && exit 1)

  # ── Deploy Production (aprobación requerida) ──────────────────────
  production:
    name:        🟢 Production
    needs:       staging
    runs-on:     ubuntu-latest
    environment: production        # ← gate de aprobación
    steps:
      - uses: azure/login@v2
        with:
          client-id:       ${{ secrets.AZURE_CLIENT_ID }}
          tenant-id:       ${{ secrets.AZURE_TENANT_ID }}
          subscription-id: ${{ secrets.AZURE_SUBSCRIPTION_ID }}
      - name: Deploy to Production
        run: |
          az containerapp update \
            --name "${{ secrets.ACA_NAME_PRD }}" \
            --resource-group "${{ secrets.RG }}" \
            --image "${{ env.ACR_NAME }}.azurecr.io/${{ env.IMAGE }}:${{ needs.build.outputs.image-tag }}"
      # Rollback instantáneo si algo falla:
      # az containerapp revision activate
      #   --name $ACA_NAME_PRD --revision [nombre-revision-anterior]
