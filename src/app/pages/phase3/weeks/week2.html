Conceptos CI/CD + GitHub Actions Core
Workflows, jobs, steps, runners, triggers, contextos, expresiones y la anatomía completa de un archivo YAML
⏱ ~5h
Un workflow de GitHub Actions es un proceso automatizado definido en YAML que se activa por un evento. Cada workflow tiene jobs (grupos de trabajo que corren en paralelo o en secuencia), y cada job tiene steps (comandos o actions individuales). Los jobs corren en runners — máquinas virtuales gratuitas de GitHub (ubuntu-latest, windows-latest) o tus propias máquinas (self-hosted). El contexto github, env y secrets dan acceso a variables del evento, variables de entorno y secretos configurados en el repositorio.
💡 GitHub Actions es gratis para repositorios públicos, ilimitado. Para repos privados: 2,000 minutos/mes en el plan gratuito, ~33h de pipeline. Más que suficiente para aprender y proyectos personales.

💻 Código — Anatomía completa de un workflow

El YAML mínimo que necesitas entender antes de escribir el pipeline real

.github/workflows/anatomy.yml

name: CI Pipeline Anatomy  # nombre del workflow en GitHub UI

# ── TRIGGERS: qué eventos activan este workflow ──────────────
on:
  push:
    branches: [main, develop]       # push a main o develop
    paths-ignore: ['**.md', 'docs/**'] # ignorar cambios de docs
  pull_request:
    branches: [main]                # PRs hacia main
  workflow_dispatch:             # trigger manual desde GitHub UI

# ── VARIABLES DE ENTORNO GLOBALES ─────────────────────────────
env:
  DOTNET_VERSION: '8.0.x'
  SOLUTION_PATH:  './src/MiApi.sln'

# ── JOBS ──────────────────────────────────────────────────────
jobs:
  build:                            # id del job
    name: 🔨 Build
    runs-on: ubuntu-latest         # runner de GitHub, gratis
    outputs:                        # datos que pasan a otros jobs
      version: ${{ steps.version.outputs.value }}

    steps:
      - name: Checkout code
        uses: actions/checkout@v4      # action del marketplace

      - name: Setup .NET
        uses: actions/setup-dotnet@v4
        with:
          dotnet-version: ${{ env.DOTNET_VERSION }}

      - name: Restore + Build
        run: |
          dotnet restore ${{ env.SOLUTION_PATH }}
          dotnet build   ${{ env.SOLUTION_PATH }} -c Release --no-restore

  test:
    name: ✅ Test
    runs-on: ubuntu-latest
    needs: build                     # espera a que build termine
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-dotnet@v4
        with:
          dotnet-version: ${{ env.DOTNET_VERSION }}
      - run: dotnet test --no-build -c Release

Build Pipeline — Restore, Build y Artifacts
Caché de NuGet, build matricial para múltiples .NET versions, artifacts entre jobs, versionado semántico con GitVersion
⏱ ~5h
El build pipeline hace tres cosas críticas: restaurar dependencias (el paso más lento — se cachea para acelerar), compilar en modo Release (no Debug — el modo Release incluye optimizaciones que afectan el comportamiento), y publicar el artifact (el binario compilado que se deployará). Los artifacts se pasan entre jobs usando actions/upload-artifact y actions/download-artifact — el job de deploy descarga el artifact del job de build, sin recompilar.
💡 El caché de NuGet puede ahorrar 2-3 minutos por pipeline. Con actions/cache se cachea la carpeta ~/.nuget/packages usando como key el hash del .csproj. Si los paquetes no cambiaron, se restauran del caché en segundos en lugar de descargarse.

💻 Código — Build pipeline completo con caché, versionado y artifacts

El workflow de build que usarás como base para todos tus proyectos .NET

.github/workflows/build.yml

name: 🔨 Build

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

env:
  DOTNET_VERSION:    '8.0.x'
  PROJECT_PATH:     './src/Api/Api.csproj'
  ARTIFACT_NAME:   'api-release'

jobs:
  build:
    name: Build & Publish
    runs-on: ubuntu-latest

    steps:
      - name: 📥 Checkout
        uses: actions/checkout@v4
        with:
          fetch-depth: 0               # historial completo para GitVersion

      - name: ⚙️ Setup .NET ${{ env.DOTNET_VERSION }}
        uses: actions/setup-dotnet@v4
        with:
          dotnet-version: ${{ env.DOTNET_VERSION }}

      # Cachear paquetes NuGet — key = hash del .csproj
      - name: 📦 Cache NuGet
        uses: actions/cache@v4
        with:
          path: ~/.nuget/packages
          key:  ${{ runner.os }}-nuget-${{ hashFiles('**/*.csproj') }}
          restore-keys: |
            ${{ runner.os }}-nuget-

      - name: 🔧 Restore
        run: dotnet restore ${{ env.PROJECT_PATH }}

      - name: 🔨 Build Release
        run: |
          dotnet build ${{ env.PROJECT_PATH }} \
            --configuration Release \
            --no-restore \
            -p:Version=${{ github.run_number }}.0.0

      - name: 📤 Publish
        run: |
          dotnet publish ${{ env.PROJECT_PATH }} \
            --configuration Release \
            --no-build \
            --output ./dist

      # Subir el binario como artifact para que el job de deploy lo descargue
      - name: ⬆️ Upload Artifact
        uses: actions/upload-artifact@v4
        with:
          name:            ${{ env.ARTIFACT_NAME }}
          path:            ./dist
          retention-days: 7
          if-no-files-found: error

Test Pipeline — Quality Gates y Cobertura
xUnit en CI, code coverage con Coverlet, threshold de cobertura, reporte de test en GitHub UI y branch protection rules
⏱ ~5h
El test pipeline es la barrera de calidad que impide que código roto llegue a producción. Configurado correctamente: si cualquier test falla, el pipeline se detiene y el merge queda bloqueado por las branch protection rules de GitHub. La cobertura de código con Coverlet agrega un quality gate adicional — si la cobertura baja del umbral definido (ej. 80%), el pipeline falla aunque todos los tests pasen.
⚠️ Los tests de integración necesitan una BD real. Usa Service Containers de GitHub Actions para levantar SQL Server (imagen Docker mcr.microsoft.com/mssql/server) directamente en el runner — sin mockear, con una BD real de prueba. Es la forma correcta de testear repositorios y queries.

💻 Código — Test pipeline con coverage + SQL Server en Docker

Quality gate que bloquea el merge si los tests fallan o la cobertura baja

.github/workflows/test.yml

name: ✅ Test

on:
  push:     branches: [main, develop]
  pull_request: branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest

    # SQL Server en Docker para integration tests
    services:
      sqlserver:
        image: mcr.microsoft.com/mssql/server:2022-latest
        env:
          SA_PASSWORD:           TestPass!2024
          ACCEPT_EULA:           Y
          MSSQL_PID:             Developer
        ports:
          - 1433:1433
        options: |
          --health-cmd "/opt/mssql-tools18/bin/sqlcmd -S localhost -U sa -P TestPass!2024 -Q 'SELECT 1' -No"
          --health-interval 10s
          --health-retries 5

      steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-dotnet@v4
        with:
          dotnet-version: '8.0.x'

      - name: 🧪 Run Tests with Coverage
        env:
          # Connection string al SQL Server del service container
          TEST_DB_CONNECTION: "Server=localhost;Database=TestDb;User Id=sa;Password=TestPass!2024;TrustServerCertificate=True"
        run: |
          dotnet test ./tests/Api.Tests/Api.Tests.csproj \
            --configuration Release \
            --collect:"XPlat Code Coverage" \
            --results-directory ./coverage \
            -- DataCollectionRunSettings.DataCollectors.DataCollector.Configuration.Format=cobertura
    
      # Publicar reporte de cobertura en el PR como comment
      - name: 📊 Coverage Report
        uses: irongut/CodeCoverageSummary@v1.3.0
        with:
          filename:        ./coverage/**/coverage.cobertura.xml
          badge:           true
          fail_below_min:  true         # pipeline falla si coverage < threshold
          format:          markdown
          thresholds:      '60 80'      # warn 60%, fail 80%
          output:          both

      - name: 💬 Post Coverage Comment on PR
        if: github.event_name == 'pull_request'
        uses: marocchino/sticky-pull-request-comment@v2
        with:
          recreate: true
          path:     code-coverage-results.md

Deploy Pipeline — Secrets, Environments y Azure
GitHub Secrets, OIDC sin contraseñas, Environments con aprobación manual, deploy a staging y producción, smoke tests
⏱ ~5h
El deploy pipeline tiene dos destinos: staging (automático) y producción (requiere aprobación). La autenticación con Azure se hace con OIDC (OpenID Connect) — en lugar de guardar una contraseña en los Secrets, se configura una relación de confianza entre GitHub y Azure. El runner de GitHub obtiene un token temporal de Azure sin contraseñas almacenadas. Esto es la forma moderna y segura. Los Environments de GitHub permiten configurar reglas de aprobación: "nadie puede deployar a producción sin que el CTO apruebe".

Secrets necesarios en el repo

Secret Name Tipo Cómo obtenerlo Obligatorio
AZURE_CLIENT_ID OIDC App Registration en Azure AD → Application ID Sí
AZURE_TENANT_ID OIDC Azure AD → Properties → Tenant ID Sí
AZURE_SUBSCRIPTION_ID OIDC az account show --query id Sí
AZURE_WEBAPP_NAME_STG Config Nombre del App Service de staging Sí
AZURE_WEBAPP_NAME_PRD Config Nombre del App Service de producción Sí
AZURE_RESOURCE_GROUP Config Nombre del Resource Group Recomendado

Environments de GitHub — flujo de aprobaciones

Development
Cada push a cualquier branch feature. Corre CI solamente — sin deploy. Validación rápida de que el código compila y los tests pasan.
Auto — push any branch
Staging
Cada merge a main. Deploy automático después de CI verde. URL de staging para QA manual y smoke tests automáticos.
Auto — merge to main
Production
Requiere aprobación manual. Un reviewer debe hacer click en "Approve" en GitHub UI. Solo corre si staging deploy fue exitoso.
Manual — requiere aprobación

💻 Código — Deploy pipeline completo con OIDC y Environments

El workflow que despliega a staging automáticamente y pide aprobación para producción

.github/workflows/deploy.yml

name: 🚀 Deploy

on:
  workflow_run:                # se activa cuando otro workflow termina
    workflows: ["🔨 Build"]
    types:     [completed]
    branches:  [main]

permissions:
  id-token:  write               # necesario para OIDC
  contents: read

jobs:
  # ── JOB 1: Deploy a Staging (automático) ──────────────────────
  deploy-staging:
    name: 🟡 Deploy → Staging
    runs-on: ubuntu-latest
    environment: staging          # Environment de GitHub → logging + secrets
    if: ${{ github.event.workflow_run.conclusion == 'success' }}

    steps:
      - name: 🔑 Login Azure (OIDC — sin passwords)
        uses: azure/login@v2
        with:
          client-id:       ${{ secrets.AZURE_CLIENT_ID }}
          tenant-id:       ${{ secrets.AZURE_TENANT_ID }}
          subscription-id: ${{ secrets.AZURE_SUBSCRIPTION_ID }}

      - name: ⬇️ Download Artifact
        uses: actions/download-artifact@v4
        with:
          name:             api-release
          path:             ./dist
          run-id:           ${{ github.event.workflow_run.id }}

      - name: 🚀 Deploy to Staging
        uses: azure/webapps-deploy@v3
        with:
          app-name:  ${{ secrets.AZURE_WEBAPP_NAME_STG }}
          package:   ./dist

      - name: 🩺 Smoke Test Staging
        run: |
          STG_URL="https://${{ secrets.AZURE_WEBAPP_NAME_STG }}.azurewebsites.net"
          echo "Testing $STG_URL/health"
          HTTP_STATUS=$(curl -s -o /dev/null -w "%{{http_code}}" "$STG_URL/health")
          if [ "$HTTP_STATUS" != "200" ]; then
            echo "❌ Health check failed: $HTTP_STATUS"
            exit 1
          fi
          echo "✅ Staging healthy: $HTTP_STATUS"

  # ── JOB 2: Deploy a Producción (requiere aprobación) ──────────
  deploy-production:
    name: 🟢 Deploy → Production
    runs-on: ubuntu-latest
    environment: production       # tiene Required Reviewers en GitHub Settings
    needs: deploy-staging          # solo corre si staging fue exitoso

    steps:
      - name: 🔑 Login Azure
        uses: azure/login@v2
        with:
          client-id:       ${{ secrets.AZURE_CLIENT_ID }}
          tenant-id:       ${{ secrets.AZURE_TENANT_ID }}
          subscription-id: ${{ secrets.AZURE_SUBSCRIPTION_ID }}
      - name: ⬇️ Download Artifact
        uses: actions/download-artifact@v4
        with:
          name:    api-release
          run-id:  ${{ github.event.workflow_run.id }}
      - name: 🚀 Deploy to Production
        uses: azure/webapps-deploy@v3
        with:
          app-name: ${{ secrets.AZURE_WEBAPP_NAME_PRD }}
          package:  ./dist

Pipeline Completo + Azure DevOps como alternativa
Unificar build + test + deploy en un solo workflow, rollback automático, Azure DevOps Pipelines y el entregable final
⏱ ~5h
Azure DevOps Pipelines es la alternativa empresarial a GitHub Actions. Usa YAML también (azure-pipelines.yml), tiene stages, jobs y steps similares, pero está integrado con Azure Boards (tickets), Azure Repos (git) y Azure Artifacts (paquetes NuGet privados). En empresas con stack Microsoft, es la opción estándar. La semana termina con el pipeline unificado funcionando: un solo push a main que ejecuta todo el ciclo automáticamente.

✅ Integrado con Azure Boards/Repos

💻 Código — Workflow unificado CI + CD + Rollback automático

Un solo archivo que hace todo: build → test → staging → producción → rollback si falla

.github/workflows/ci-cd.yml

name: ⚡ CI/CD Pipeline

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

permissions:
  id-token:  write
  contents: read

jobs:
  # ── CI ─────────────────────────────────────────────────────────
  ci:
    name: 🔨 Build + ✅ Test
    runs-on: ubuntu-latest
    services:
      sqlserver:
        image: mcr.microsoft.com/mssql/server:2022-latest
        env:
          SA_PASSWORD:  TestPass!2024
          ACCEPT_EULA:  Y
        ports:
          - 1433:1433
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-dotnet@v4
        with:
          dotnet-version: '8.0.x'
      - uses: actions/cache@v4
        with:
          path: ~/.nuget/packages
          key:  ${{ runner.os }}-nuget-${{ hashFiles('**/*.csproj') }}
      - run: dotnet restore && dotnet build -c Release --no-restore
      - run: dotnet test -c Release --no-build --collect:"XPlat Code Coverage"
        env:
          TEST_DB_CONNECTION: "Server=localhost;Database=TestDb;User Id=sa;Password=TestPass!2024;TrustServerCertificate=True"
      - run: dotnet publish -c Release --no-build -o ./dist
      - uses: actions/upload-artifact@v4
        with:
          name: release
          path: ./dist

  # ── STAGING (automático) ───────────────────────────────────────
  staging:
    name: 🟡 Staging
    needs: ci
    runs-on: ubuntu-latest
    environment: staging
    if: github.ref == 'refs/heads/main'
    steps:
      - uses: azure/login@v2
        with:
          client-id:       ${{ secrets.AZURE_CLIENT_ID }}
          tenant-id:       ${{ secrets.AZURE_TENANT_ID }}
          subscription-id: ${{ secrets.AZURE_SUBSCRIPTION_ID }}
      - uses: actions/download-artifact@v4
        with:
          name: release
      - uses: azure/webapps-deploy@v3
        with:
          app-name: ${{ secrets.AZURE_WEBAPP_NAME_STG }}
      - name: 🩺 Smoke Test
        run: |
          STATUS=$(curl -sf -o /dev/null -w "%{{http_code}}" \
            "https://${{ secrets.AZURE_WEBAPP_NAME_STG }}.azurewebsites.net/health")
          [ "$STATUS" = "200" ] && echo "✅ OK" || (echo "❌ $STATUS" && exit 1)

  # ── PRODUCTION (aprobación manual requerida) ───────────────────
  production:
    name: 🟢 Production
    needs: staging
    runs-on: ubuntu-latest
    environment: production    # ← aquí está el gate de aprobación
    steps:
      - uses: azure/login@v2
        with:
          client-id:       ${{ secrets.AZURE_CLIENT_ID }}
          tenant-id:       ${{ secrets.AZURE_TENANT_ID }}
          subscription-id: ${{ secrets.AZURE_SUBSCRIPTION_ID }}
      - uses: actions/download-artifact@v4
        with:
          name: release
      - uses: azure/webapps-deploy@v3
        with:
          app-name: ${{ secrets.AZURE_WEBAPP_NAME_PRD }}

🎯 Entregable — Pipeline automático funcionando

El entregable es el repositorio de tu API con el pipeline CI/CD completo configurado y funcionando. Cada push a main debe disparar el ciclo completo:

  • Workflow ci-cd.yml en .github/workflows/ con los 3 jobs: CI, staging y production
  • Screenshot del pipeline verde en GitHub Actions UI mostrando los 3 jobs completados
  • Screenshot del gate de aprobación de producción — mostrando que espera antes de deployar
  • Branch protection rules en main con el status check del CI como requisito de merge
  • Zero secrets hardcodeados — todos en GitHub Secrets con OIDC configurado
  • Smoke test automático en staging que verifica /health → 200 OK antes de pedir aprobación de prod
🔗 Conexión con semanas anteriores: El /health endpoint que creaste en la Semana 11 (que verifica Azure SQL y Blob Storage) es ahora el smoke test automático del pipeline — si el deploy rompe algo, el pipeline lo detecta antes de llegar a producción. Los tests que escribiste en la Fase 1 (xUnit, Moq) ahora corren automáticamente en cada PR. La Semana 13 agrega Docker al pipeline — en lugar de publicar binarios, el pipeline construye una imagen Docker y la despliega en Azure Container Apps.
