Code Smells — Reconocer el problema antes de tocarlo
El catálogo de Martin Fowler, los 8 smells más comunes en código .NET, cómo identificarlos con análisis estático (SonarQube, Roslyn analyzers), y métricas de código objetivas
⏱ ~4h
Un code smell no es un bug — es una señal de que algo en el diseño puede mejorarse. Martin Fowler definió el término en su libro "Refactoring" (1999, 2018): son indicios en el código de que probablemente hay un problema de diseño más profundo. El procedimiento correcto antes de refactorizar tiene tres pasos: primero identificar el smell (¿qué huele mal y por qué?), luego asegurar cobertura de tests (sin tests, refactorizar es cambiar el comportamiento a ciegas), y finalmente aplicar la técnica específica que Fowler recomienda para ese smell. Sin ese orden, "refactorizar" es solo reescribir código que funciona y romper cosas.
⚠️ La regla de oro del refactoring: los tests tienen que pasar antes y después. Si no tienes tests antes de empezar, la primera tarea no es refactorizar — es escribir tests de caracterización que documenten el comportamiento actual. Luego refactorizas. Si los tests pasan antes y fallan después, revertiste un comportamiento. Si los tests pasan en ambos lados, refactorizaste correctamente.

Técnicas de Refactoring de Fowler
Extract Method, Extract Class, Replace Conditional with Polymorphism, Introduce Parameter Object, Replace Primitive with Value Object, y Strangler Fig a nivel de clase
⏱ ~5h
Fowler cataloga 66 técnicas de refactoring en su libro — estas 6 resuelven el 80% de los smells que encontrarás en código .NET real. Cada técnica tiene una mecánica concreta y reproducible: no son opiniones de diseño, son operaciones con pasos definidos. Extract Method toma un fragmento de código, le da nombre, y lo convierte en un método propio. Replace Primitive with Value Object toma un string email y lo convierte en una clase Email con su propia validación. La clave: hacer una técnica a la vez, ejecutar los tests entre cada paso, y hacer commit después de cada técnica exitosa. Commits pequeños y frecuentes — si algo falla, el revert es trivial.
Técnica Cuándo aplicarla Resultado
Extract Method Fragmento de código con un comentario encima que explica qué hace — el comentario es el nombre del método. Método con nombre expresivo, testeable de forma independiente
Extract Class Una clase con dos grupos de datos y métodos que no se relacionan entre sí — tiene dos responsabilidades. Dos clases cohesivas, cada una con una única responsabilidad
Replace Primitive with Value Object string email o decimal amount viajando por la app sin validación. La validación se duplica en cada uso. Clase Email o Money que valida en el constructor — imposible crear inválidos
Replace Conditional with Polymorphism switch (order.Type) {{'{'}} case "Premium": ... {{'}'}} repetido en 5 métodos diferentes del servicio. Clase base Order con subclases — cada tipo implementa su propia lógica sin condicionales
Introduce Parameter Object Método con 5+ parámetros, especialmente si siempre van juntos en múltiples llamadas. Record o clase CreateOrderRequest con los parámetros agrupados
Move Method / Move Field Un método que usa más datos de otra clase que de la suya propia (Feature Envy). La lógica vive donde están los datos — elimina dependencia y acoplamiento

Mejora de Arquitectura — Evolucionar sin romper
Anti-patrones arquitecturales comunes en .NET, Strangler Fig a nivel de módulo, Feature Flags para refactorings progresivos, separar la capa de datos del dominio
⏱ ~5h
Los anti-patrones arquitecturales son code smells a escala de sistema — más costosos y más difíciles de revertir. Los tres más comunes en proyectos .NET: el Anemic Domain Model (entidades que son solo bolsas de datos con getters/setters, toda la lógica en los Services — viola el principio de objeto), el Big Ball of Mud (sin capas claras, todo llama a todo, imposible de testear), y la Shared Database en microservicios (dos servicios que comparten la misma tabla — acoplamiento máximo disfrazado de practicidad). El patrón de migración arquitectural por excelencia es el Strangler Fig: envolver gradualmente el código viejo con el nuevo, redirigir tráfico progresivamente, hasta que el código viejo muere solo.
✅ Feature Flags para refactorings de alto riesgo. Cuando refactorizas un componente crítico, en lugar de hacer un big-bang release, usa un feature flag: el 5% del tráfico va a la nueva implementación, el 95% sigue con la vieja. Si la nueva falla, apagas el flag en 30 segundos sin deploy. Azure App Configuration y la librería Microsoft.FeatureManagement dan este patrón en minutos.

ADRs y Documentación Técnica Profesional
Architecture Decision Records, README técnicos que añaden valor real, diagramas C4 con Mermaid, comentarios de código que explican el porqué — no el qué
⏱ ~4h
La documentación técnica más valiosa no explica qué hace el código — eso lo hace el código mismo. Explica por qué se tomó cada decisión. El Architecture Decision Record (ADR) es el documento que responde "¿por qué usamos YARP y no Ocelot?", "¿por qué SQL Server y no PostgreSQL?", "¿por qué RabbitMQ y no Azure Service Bus?". Sin ADRs, ese conocimiento vive solo en la cabeza de quien tomó la decisión — y cuando esa persona deja el proyecto, se pierde. Un ADR tiene cinco secciones: contexto, decisión, estado (proposed/accepted/deprecated), consecuencias positivas, y consecuencias negativas. Se escribe en el momento de la decisión y se versiona en Git junto al código.
ADR-007: Usar YARP como API Gateway en lugar de Ocelot
Contexto
Necesitamos un API Gateway para enrutar tráfico a 4 microservicios, agregar autenticación JWT, y servir documentación Swagger unificada. Los candidatos evaluados fueron YARP, Ocelot, y Azure API Management.
Decisión
Usamos YARP (Yet Another Reverse Proxy) de Microsoft. Es el único mantenido activamente por Microsoft en .NET 8+, integra nativamente con el ecosistema ASP.NET Core (middlewares, DI, configuration), y tiene soporte de primera clase en Azure Container Apps.
Consecuencias
Positivo: configuración via appsettings.json, actualizaciones de rutas sin rebuild, soporte completo de OpenTelemetry. Negativo: menos plugins out-of-the-box que Ocelot para throttling avanzado — implementamos rate limiting con el middleware nativo de .NET en su lugar.

Pull Request Profesional — Comunicar el cambio
Anatomía de un PR que se aprueba rápido, descripción de cambios con contexto, checklist de revisión, convenciones de commit (Conventional Commits), y el entregable final
⏱ ~4h
Un Pull Request es un documento de comunicación, no solo una lista de cambios de código. El PR ideal responde tres preguntas en los primeros 30 segundos: ¿qué problema resuelve?, ¿cómo lo resuelve?, y ¿cómo verificar que funciona?. Un PR con título "fix stuff" y sin descripción es una señal de que el autor no respeta el tiempo del revisor. Las convenciones de Conventional Commits hacen que el historial de Git sea legible como un changelog: feat(orders): add order cancellation with reason, refactor(domain): extract Email value object, fix(auth): token revocation not persisting in Redis. Herramientas como semantic-release generan changelogs automáticos desde el historial de commits.

🎯 Entregable — Codebase refactorizado con documentación viva

El entregable es el proyecto del roadmap con mejoras concretas de calidad, documentado de forma que un nuevo desarrollador pueda entenderlo en un día:

  • Al menos 3 smells eliminados: uno con Extract Method, uno con Value Object, y uno con Extract Class o Move Method
  • Carpeta /docs/adr/ con mínimo 2 ADRs — una decisión de tecnología y una decisión de arquitectura
  • README con diagrama C4 Container en Mermaid que muestre todos los servicios del roadmap y sus relaciones
  • Historial de commits de la semana en formato Conventional Commits — refactor, feat, docs, test
  • Un PR real (aunque sea en un fork propio) con la plantilla completa — descripción, cambios, verificación y checklist
  • Screenshot de SonarLint o Roslyn Analyzers mostrando 0 issues Critical en el código refactorizado
🔗 La semana que cierra la fase 4: Esta semana aplica el criterio de calidad a todo el trabajo anterior. Los Value Objects mejoran el dominio de la Semana 17. Los ADRs documentan las decisiones de seguridad de la Semana 19 y observabilidad de la Semana 20. El diagrama C4 unifica visualmente toda la arquitectura construida. La semana 21 no agrega tecnología nueva — agrega intención, claridad y sostenibilidad al código que ya existe.
