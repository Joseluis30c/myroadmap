Las 4 estrategias de versionado — cuándo usar cada una

URL Segment
/api/v2/orders
Más visible
La versión está en la URL. Todo el mundo la ve, es fácil de testear con el browser y cURL. El estándar de facto para APIs públicas.
Recomendado
Query String
?api-version=2.0
Más flexible
La versión va en el query string. Útil cuando no puedes cambiar la estructura de la URL. Menos limpio pero funcional.
Opcional
HTTP Header
api-version: 2.0
Más limpio
La versión va en el header. Las URLs son inmutables entre versiones. Difícil de testear con el browser sin herramientas.
APIs internas
Media Type

Más puro
La versión está en el Accept header usando content negotiation. Correcto en teoría REST, pero complejo en práctica. GitHub lo usa.

API Versioning — Por qué y cómo
El problema del breaking change, las 4 estrategias de versionado, semver para APIs, política de deprecación y sunset headers
⏱ ~4h
Un breaking change es cualquier cambio que rompe el código de un cliente existente. Renombrar un campo JSON, eliminar un campo, cambiar el tipo de un parámetro — todos rompen a los clientes que ya están en producción. El versionado no es para ti: es para tus clientes. Les da tiempo de migrar mientras siguen usando la versión anterior. La regla de oro: nunca elimines ni renombres campos en una versión existente. Agrega nuevos campos (backward compatible), y cuando necesites breaking changes, crea una versión nueva.
⚠️ Breaking changes más comunes que olvidan versionar: cambiar el nombre de un campo JSON (userId → user_id), cambiar un campo de string a object, eliminar un endpoint, cambiar el significado de un código HTTP (de 200 a 204), y cambiar la estructura de los errores. Agregar campos opcionales generalmente no es breaking — pero eliminar o renombrar siempre lo es.
Tipo de cambio ¿Breaking? ¿Requiere nueva versión? Ejemplo
Agregar campo opcional NO No necesario + "discount": 0.1
Renombrar campo SÍ Obligatorio userId → user_id
Eliminar campo SÍ Obligatorio - "legacyCode"
Cambiar tipo de campo SÍ Obligatorio string → number
Agregar campo requerido en request SÍ Obligatorio + required: "taxId"
Nuevo endpoint NO No necesario GET /orders/{{'{'}}id{{'}'}}/timeline
Cambiar código HTTP SÍ Obligatorio 200 → 204 en DELETE

Asp.Versioning — Implementación en .NET 8
Configurar Asp.Versioning, versionar controllers y minimal APIs, múltiples estrategias simultáneas, deprecar versiones y sunset headers automáticos
⏱ ~4h
Asp.Versioning es el paquete oficial de Microsoft para versionar APIs en ASP.NET Core. Reemplaza al antiguo Microsoft.AspNetCore.Mvc.Versioning que quedó obsoleto. Con una sola configuración en Program.cs soporta simultáneamente URL segment (/v1/), query string (?api-version=1.0) y header (api-version: 1.0). El atributo [ApiVersion] en el controller declara qué versiones maneja, y [MapToApiVersion] permite que un mismo controller tenga distintas implementaciones por versión.
💡 Sunset headers — notifica a los clientes antes de eliminar una versión. El header Sunset: Sat, 31 Dec 2025 23:59:59 GMT le dice al cliente que esta versión va a dejar de funcionar en esa fecha. Asp.Versioning los agrega automáticamente cuando marcas una versión como deprecated. Los clientes bien construidos loggean una advertencia cuando reciben este header — es la forma profesional de gestionar el ciclo de vida de una API.

Documentación Real — El nivel Stripe
Problem Details RFC 7807, errores consistentes y documentados, changelogs, guías de migración entre versiones, README de API y generación de SDKs con NSwag
⏱ ~4h
La documentación real va más allá del Swagger automático. Stripe tiene la mejor documentación de API del mundo no porque use herramientas mejores, sino porque invierte en tres cosas: errores consistentes y descriptos (todos los errores tienen el mismo formato con un código de error específico y una descripción accionable), guías de migración (un documento por cada versión mayor que explica exactamente qué cambió y cómo migrar), y ejemplos reales en múltiples lenguajes. El estándar para errores es RFC 7807 — Problem Details — que ASP.NET Core implementa con ProblemDetails.
💡 Problem Details (RFC 7807) es el estándar para errores de API. En lugar de retornar un JSON ad-hoc o solo el código HTTP, retornas un objeto con type (URL que documenta el error), title (descripción legible), status (código HTTP), detail (descripción específica del caso) y opcionalmente instance (URL de esta ocurrencia específica). ASP.NET Core lo implementa con AddProblemDetails().

🎯 Entregable — API versionada con documentación profesional

El entregable de la semana es una API con al menos dos versiones activas, documentación completa y errores estandarizados:

  • Al menos un endpoint con dos versiones: GET /api/v1/orders/{{'{'}}id{{'}'}} y GET /api/v2/orders/{{'{'}}id{{'}'}} retornando modelos distintos
  • v1 marcada como deprecated con X-Api-Deprecated-Versions y Sunset en los response headers
  • Swagger UI con selector de versión — al cambiar de v1 a v2 se ven documentos distintos con los endpoints correspondientes
  • Botón Authorize funcional — ejecutar un endpoint autenticado desde Swagger sin Postman
  • Errores siguen Problem Details — un 404, un 422 y un 401 tienen el mismo formato consistente
  • Screenshot del Swagger en el Gateway mostrando los documentos de todos los microservicios en el dropdown
🔗 Conexión con la Fase 4: El versionado y la documentación de esta semana se construyen sobre la separación de microservicios de la Semana 17. Cada servicio tiene su propia versión y su propio Swagger — el Gateway los agrega en una sola UI. La Semana 19 introduce CQRS con MediatR dentro de cada servicio, y la Semana 20 cierra con el Outbox Pattern y Domain Events — donde el versionado del contrato de eventos (no solo HTTP) también será relevante.
