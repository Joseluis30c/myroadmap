OAuth 2.0 y OpenID Connect
Flows: Authorization Code, Client Credentials, Device. Diferencia OAuth vs OIDC. Scopes, claims, tokens. Keycloak local con Docker
⏱ ~5h
OAuth 2.0 es un protocolo de autorización — OIDC (OpenID Connect) es la capa de autenticación encima de OAuth. OAuth responde "¿qué puede hacer esta app?", OIDC responde "¿quién es este usuario?". La confusión más común: OAuth no fue diseñado para autenticación — fue diseñado para que una app acceda a recursos en nombre del usuario sin que el usuario le entregue su contraseña. OIDC agrega el concepto de identity token (JWT con info del usuario) sobre el access token de OAuth. Los scopes definen qué puede acceder el token (orders:read, orders:write), y los claims son los datos del usuario dentro del JWT (sub, email, role).

JWT Avanzado — Lo que falla en producción
Rotación de claves, token refresh, revocación con Redis, almacenamiento seguro en cliente, el algoritmo RS256 vs HS256 y JWT best practices
⏱ ~5h
El 90% de las implementaciones JWT tienen al menos uno de estos tres errores. Primero: algorithm: none — algunos servidores aceptan tokens sin firma si el header dice "alg":"none"; siempre validar el algoritmo explícitamente. Segundo: secret demasiado corto — con HS256 y un secret de 8 caracteres, un atacante puede crackearlo offline en minutos; mínimo 256 bits. Tercero: no hay revocación — una vez emitido un JWT, es válido hasta que expira aunque el usuario cierre sesión; necesitas una lista de tokens revocados en Redis o tokens de corta duración con refresh tokens. La recomendación actual: RS256 (asimétrico, par de claves pública/privada) en lugar de HS256 para APIs públicas.
⚠️ Nunca stores el access token en localStorage. localStorage es accesible por cualquier script JavaScript en la página — un ataque XSS lo roba en una línea. La alternativa segura: httpOnly cookies para el refresh token (inaccesible para JS), y access tokens de corta duración (15 min) en memoria del cliente. Si necesitas un SPA, investiga el BFF (Backend for Frontend) pattern.

OWASP Top 10 API Security — Ataques y defensas en .NET
SQL Injection con EF Core, IDOR, Mass Assignment, Security Misconfiguration, CORS correcto, headers de seguridad y validación de input con FluentValidation
⏱ ~5h
OWASP API Security Top 10 es distinto al OWASP Web Top 10. Las APIs tienen ataques específicos: Broken Object Level Authorization (IDOR) es el #1 — el usuario 42 cambia el ID en la URL y accede al pedido del usuario 43. Broken Object Property Level Authorization es el #3 — Mass Assignment: el cliente envía isAdmin: true en el body del POST y el objeto lo mapea sin filtrar. Unrestricted Resource Consumption es el #4 — sin rate limiting ni límites de paginación, alguien puede pedir ?pageSize=99999. En ASP.NET Core, el mayor aliado es FluentValidation para inputs, [JsonIgnore] o records inmutables para Mass Assignment, y ResourceOwnershipFilter personalizado para IDOR.

Rate Limiting — Protección contra abuso y DoS
Fixed window, sliding window y token bucket, RateLimiter nativo de .NET 7+, rate limiting por IP/usuario/endpoint, y Azure API Management como alternativa cloud
⏱ ~4h
.NET 7 introdujo System.Threading.RateLimiting — rate limiting nativo sin librerías externas. Hay cuatro algoritmos: Fixed Window (100 requests por minuto, el contador se reinicia al minuto), Sliding Window (100 requests en los últimos 60 segundos móviles — más justo, evita ráfagas en el límite de ventana), Token Bucket (tokens que se recargan gradualmente — permite ráfagas cortas), y Concurrency Limiter (máximo N requests simultáneas, no por tiempo). Para APIs públicas el estándar es sliding window por IP. Para APIs autenticadas, por userId del JWT para que un usuario abusador no afecte a otros.
Fixed Window
Ventana fija
El contador se reinicia cada ventana. Permite ráfagas justo en el límite. Más simple de implementar.
riesgo de burst
Sliding Window
Ventana deslizante
Considera los últimos N segundos móviles. Sin ráfagas en límites de ventana. Más justo y uniforme.
recomendado

Cubo de tokens

Hardening Completo — Audit, HTTPS, CORS y checklist de producción
Audit log de operaciones sensibles, HTTPS enforce, CORS correcto para microservicios, Data Protection API para cookies y el entregable final
⏱ ~4h
El hardening es la diferencia entre una API que funciona y una API que es segura en producción. Cinco cambios de alta prioridad que se olvidan frecuentemente: activar UseHttpsRedirection y UseHsts, configurar CORS con orígenes explícitos (nunca AllowAnyOrigin con AllowCredentials), deshabilitar Swagger en producción con if (app.Environment.IsDevelopment()), activar el Audit Log para operaciones sensibles (quién creó/borró un pedido, con qué IP y cuándo), y configurar Data Protection API para que las claves de cifrado persistan entre reinicios del container.
✅ Checklist de seguridad antes de cada deploy a producción: HTTPS + HSTS activados, Swagger deshabilitado, CORS con lista blanca de orígenes, stack traces ocultos en errores, secrets en Key Vault (no en appsettings.json), Rate Limiting activado, security headers presentes, logs sin datos sensibles (passwords, tokens, PII), y validación de input en todos los endpoints con FluentValidation.

🎯 Entregable — API con hardening completo de producción

El entregable de esta semana es una API que supera el checklist de seguridad antes de ir a producción real:

  • OAuth 2.0 funcional: endpoint protegido que solo acepta tokens con el scope correcto emitidos por el auth server
  • Demostrar IDOR bloqueado: request con JWT del usuario A intentando acceder al recurso del usuario B retorna 403
  • Rate limiting en acción: screenshot de la respuesta 429 con el header Retry-After en el cuerpo Problem Details
  • Logout que revoca el token: petición con el token revocado retorna 401 aunque el token aún no haya expirado
  • Security headers verificados con Mozilla Observatory — puntuación mínima B+
  • Audit Log en Application Insights: query KQL que muestra quién borró qué recurso y desde qué IP
🔗 Conexión con la Fase 4: La seguridad de esta semana se aplica a los microservicios de la Semana 17: el API Gateway centraliza la autenticación (un solo punto de validación JWT) y el rate limiting global, mientras cada microservicio sigue validando autorización a nivel de recurso (IDOR). El OAuth configurado aquí es la base de la Semana 20 donde los eventos de dominio también necesitan autenticación service-to-service con Client Credentials flow.
