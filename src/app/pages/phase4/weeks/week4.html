Los 3 pilares de la observabilidad — qué es cada uno

📋
Logging
Serilog + Seq
Registro de eventos discretos con contexto estructurado. No es Console.WriteLine — es un evento JSON con campos indexables: OrderId, UserId, Duration. Permite filtrar y correlacionar.
📊
Metrics
App Insights + Azure Monitor
Series temporales numéricas: requests por segundo, latencia P99, tasa de errores, uso de CPU/memoria. Permiten alertas basadas en umbrales y dashboards en tiempo real.
🔍
Tracing
Distributed Tracing
Seguir una request a través de múltiples microservicios usando un TraceId común. Un request que pasa por Gateway → Orders → Catalog → BD queda trazado de inicio a fin.

Serilog — Setup completo desde cero
Instalar y configurar Serilog en ASP.NET Core 8, sinks (Console, File, Application Insights), enrichers, mínimum level por namespace y appsettings.json driven config
⏱ ~4h
Serilog es el estándar de facto para logging en .NET — la diferencia clave es que escribe objetos, no strings. El logging tradicional hace logger.Log($"Order {{'{'}}order.Id{{'}'}} created for {{'{'}}order.UserId{{'}'}}") — eso produce texto plano que no puedes filtrar. Serilog con structured logging hace logger.Information("Order {{'{'}}OrderId{{'}'}} created for {{'{'}}UserId{{'}'}}", order.Id, order.UserId) — eso produce un JSON con campos OrderId y UserId que Application Insights indexa y hace queryable con KQL. La diferencia en producción: con texto plano tardas 30 minutos buscando un bug; con logging estructurado tardas 30 segundos con un filtro.
Level Cuándo usarlo Entornos recomendados
Verbose Cada detalle: entrada y salida de métodos, valores de variables. Solo para debugging profundo. development (opcional)
Debug Estado de la aplicación que ayuda a diagnosticar problemas. Cache hits/misses, queries SQL, decisiones de routing. development + staging
Information El flujo normal del negocio: orden creada, usuario autenticado, mensaje publicado. Lo que importa en producción. todos
Warning Algo inesperado que no falló, pero podría causar problemas: query lenta, retry exitoso, configuración inusual. todos
Error Operación fallida que el sistema pudo manejar: excepción capturada, request rechazada, recurso no encontrado. todos → alerta
Fatal Error que hace caer la aplicación. El sistema no puede continuar operando. Siempre genera alerta urgente. todos → alerta crítica

Structured Logging — Contexto que hace útil cada log
Correlation ID entre microservicios, LogContext para enriquecer scopes, destructuring de objetos, evitar logging de datos sensibles y el patrón de logging en Clean Architecture
⏱ ~5h
El structured logging vale cero si no agregas el contexto correcto. El principio: cada log de negocio debe responder qué pasó, con qué entidad y para qué usuario. Cuando llega una excepción en producción, la pregunta es: "¿qué estaba haciendo el usuario en ese momento?" — si el log dice Error: NullReferenceException sin contexto, no puedes responderla. Si dice Error: NullReferenceException | OrderId=a1b2 | UserId=user_42 | TraceId=abc123, puedes reconstruir exactamente qué pasó en 2 minutos. El Correlation ID es el hilo que une todos los logs de un mismo request a través de múltiples microservicios.
⚠️ Nunca loggees datos sensibles. Passwords, tokens JWT, números de tarjeta de crédito, o cualquier PII (Personally Identifiable Information) en los logs es un problema de compliance (GDPR) y de seguridad. Serilog tiene destructuring policies para enmascarar campos automáticamente. Si loggas una entidad User, asegúrate de que el PasswordHash y RefreshToken estén excluidos con [LogMasked] o una política personalizada.

Azure Monitor + Application Insights + KQL
Configurar Application Insights en .NET 8, las 5 tablas más importantes, queries KQL esenciales para responder preguntas de producción, Live Metrics y Application Map
⏱ ~5h
Application Insights es la herramienta de observabilidad de Azure — el backend donde van a parar todos tus logs de Serilog, métricas de ASP.NET Core y trazas de excepciones. Su lenguaje de consulta es KQL (Kusto Query Language) — SQL pero para datos de telemetría. Las 5 tablas que usarás el 90% del tiempo: requests (todas las peticiones HTTP con latencia y status code), exceptions (todas las excepciones con stack trace), traces (tus logs de Serilog), dependencies (llamadas a SQL Server, Redis, servicios externos) y customMetrics (métricas de negocio que defines tú). Application Map muestra visualmente cómo fluyen las llamadas entre tus microservicios — indispensable para identificar cuellos de botella.

Health Checks + Alertas en Azure
ASP.NET Core Health Checks para SQL Server, Redis y servicios externos, endpoint /health, Azure Monitor Alerts con condiciones KQL y notificaciones por email/Teams
⏱ ~4h
Los health checks son el sistema nervioso de tus microservicios en producción. El endpoint /health responde si el servicio está listo para recibir tráfico — Azure Container Apps lo usa para reiniciar pods que fallan (liveness probe) y para no enviar tráfico a pods que todavía están iniciando (readiness probe). Un health check completo verifica: conexión a la base de datos, conexión a Redis, espacio en disco, y dependencias críticas. El estándar: devolver 200 Healthy, 200 Degraded (funciona pero con problemas), o 503 Unhealthy (no apto para recibir tráfico).
Liveness Probe
¿Está vivo?
Si falla, el orchestrador (Container Apps, Kubernetes) reinicia el pod. Solo chequea que el proceso responde, no sus dependencias.
GET /health/live
Readiness Probe
¿Listo para tráfico?
Si falla, el load balancer deja de enviarle tráfico pero no reinicia el pod. Chequea BD, Redis y dependencias externas.
GET /health/ready
Startup Probe
¿Terminó de iniciar?
Da tiempo al servicio para ejecutar migraciones o cargar caché antes de empezar a recibir tráfico. Temporal durante el startup.
GET /health/startup

Manejo de Errores en Producción
Exception middleware global, exception filters, IExceptionHandler (.NET 8), runbooks de incidentes, alertas en Azure Monitor, y el entregable final de la semana
⏱ ~4h
.NET 8 introduce IExceptionHandler — la forma moderna de centralizar el manejo de excepciones en una API. El patrón correcto: cada tipo de excepción de dominio (NotFoundException, ValidationException, UnauthorizedException) tiene su handler que la convierte en la respuesta HTTP y Problem Details correctos. Las excepciones no capturadas llegan a un handler de último recurso que loggea con toda la información disponible (TraceId, UserId, request path, request body si es seguro) y devuelve un 500 genérico sin stack trace. El objetivo: nunca mostrar información interna al cliente, pero siempre tener suficiente contexto en los logs para diagnosticar.
✅ El runbook de incidentes es el documento que defines HOY para usar a las 3am. Para cada tipo de alerta (error rate > 5%, latencia P99 > 2s, health check unhealthy), escribe: 1) qué significa, 2) cómo verificar el impacto con KQL, 3) los pasos de mitigación inmediata (roll back, scale out, restart), y 4) cómo encontrar la causa raíz. Un runbook bien escrito reduce el MTTR (Mean Time to Recover) de horas a minutos.

🎯 Entregable — API completamente observable en producción

El entregable es una API que puedes operar con confianza en producción — donde cada incidente tiene respuesta en minutos:

  • Screenshot de Application Insights con logs de Serilog que muestran el Correlation ID propagado a través de 2 microservicios
  • 3 queries KQL ejecutadas en el portal de Azure: excepciones por tipo, latencia P99 por endpoint, y trazado de un request específico
  • Endpoint /health con respuesta JSON mostrando el estado de SQL Server, Redis y la API dependiente
  • Alerta en Azure Monitor configurada: dispara cuando error rate > 5% en los últimos 5 minutos — notificación visible
  • Demostrar que una excepción no manejada devuelve Problem Details sin stack trace y el requestId aparece en los logs de App Insights
  • Application Map en App Insights mostrando las dependencias de tu API (SQL Server, Redis, microservicio externo)
🔗 Conexión con semanas anteriores: La observabilidad conecta todo lo anterior: los logs de Serilog capturan los eventos del dominio (Sem 17-18), las excepciones manejan los errores de seguridad con Problem Details (Sem 19), y Application Insights da visibilidad de los mensajes de RabbitMQ/Service Bus (Sem 16) y las respuestas del cache Redis (Sem 15). Un sistema observable es un sistema que puedes operar con confianza.
