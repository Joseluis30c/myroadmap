1
CTEs Avanzados
CTEs múltiples encadenados, recursivos para jerarquías y cuándo usar temp table en lugar de CTE
⏱ ~5h
Un CTE no es magia de performance. En SQL Server, un CTE no se materializa por defecto — se expande inline como una subquery. Lo que ganas es legibilidad y la capacidad de referenciar el mismo resultado múltiples veces. El CTE recursivo tiene valor semántico único: permite recorrer estructuras jerárquicas (org charts, categorías padre-hijo, BOM) que serían imposibles de expresar con SQL plano.

Anatomía de un CTE recursivo

Anchor Member (caso base)
El punto de partida. Selecciona las filas raíz (empleados sin jefe, categorías de nivel 0). Se ejecuta UNA sola vez.
↓ UNION ALL ↓
Recursive Member (paso iterativo)
Se une al CTE mismo para ir un nivel más abajo en la jerarquía. Se ejecuta repetidamente hasta que no retorna filas.
↓ RESULT ↓
Query final sobre el CTE
Usa el CTE como si fuera una tabla. Filtra, ordena o agrega los datos de todos los niveles de la jerarquía.
⚠️ MAXRECURSION: El default es 100 niveles. Si tu jerarquía puede ser más profunda, agrega OPTION (MAXRECURSION 0). Sin esto, una jerarquía circular lanzará un error.

2
Window Functions
OVER, PARTITION BY, frame clauses, funciones de ranking y analytics sin colapsar filas
⏱ ~5h
Las window functions son la herramienta más poderosa y subutilizada del SQL moderno. A diferencia de GROUP BY, no colapsan filas — calculan sobre una "ventana" de filas relacionadas manteniendo el detalle. Esto elimina self-joins y subconsultas correlacionadas que son lentas por naturaleza. Un SUM() OVER (PARTITION BY ...) reemplaza un subquery correlacionado con un solo escaneo.
SUM(Salary) OVER (PARTITION BY DepartmentId ORDER BY HireDate ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW)
Función
SUM, AVG, COUNT, MIN, MAX, ROW_NUMBER, RANK, DENSE_RANK, LAG, LEAD, NTILE, FIRST_VALUE, LAST_VALUE
PARTITION BY
Reinicia el cálculo para cada grupo. Equivalente al GROUP BY pero sin colapsar filas. Opcional.
ORDER BY
Define el orden dentro de cada partición. Requerido para funciones de ranking y LAG/LEAD.
Frame Clause
Define qué filas forman la ventana. ROWS vs RANGE, UNBOUNDED PRECEDING, CURRENT ROW, N PRECEDING.

Optimización de JOINs
Nested Loops vs Hash Match vs Merge Join, conversiones implícitas, subconsultas correlacionadas y APPLY
⏱ ~5h
SQL Server elige el algoritmo de join automáticamente basándose en estadísticas. Nested Loops es ideal cuando la tabla exterior es pequeña y la interior tiene índice. Hash Match cuando no hay índices útiles. Merge Join cuando ambas entradas ya están ordenadas. Entender cuándo cada algoritmo es correcto te permite diagnosticar cuándo el optimizer tomó una mala decisión — y cuál es la causa raíz.

Los 3 algoritmos de JOIN

Algoritmo Complejidad Cuándo el optimizer lo elige Señal de problema
Nested Loops O(n · log m) Tabla exterior pequeña + índice en columna JOIN de la tabla interior Sin índice en la FK → O(n · m), catastrófico
Hash Match O(n+m) Sin índice útil, tablas grandes, cardinalidad alta. Requiere memoria. Memory Spill a TempDB si estimación es mala
Merge Join O(n+m) Ambas entradas ya ordenadas por la columna JOIN (índice o Sort explícito) Sort explícito agrega costo si no hay índice

Estadísticas
Histogramas, density vectors, parameter sniffing y cómo las estadísticas obsoletas destruyen planes óptimos
⏱ ~5h
El optimizador trabaja con estadísticas, no con los datos reales. Para elegir entre Index Seek vs Scan, Nested Loops vs Hash Match, necesita estimar cuántas filas retornará cada operación. Si las estadísticas están desactualizadas, esas estimaciones son incorrectas y el plan resultante puede ser catastrófico: estimó 1 fila, obtuvo 900,000 — eligió Nested Loops y tardó 8 minutos.

💡 Parameter Sniffing: SQL Server compila el plan de un stored procedure con los parámetros de la primera ejecución y lo reutiliza. Si la primera llamada fue con un valor atípico, el plan es ineficiente para todos los demás. Soluciones: OPTION (RECOMPILE), OPTIMIZE FOR UNKNOWN, o local variables.

Queries Complejas Reales
Temp tables vs CTEs, paginación eficiente, keyset pagination y el workflow completo de tuning
⏱ ~5h
Las temp tables materializan el resultado intermedio — esto es lo que el CTE no hace. Para queries de múltiples pasos donde el optimizador toma malas decisiones al encadenar CTEs (no puede ver las estadísticas del CTE anterior), una tabla temporal con índices te da control total. Es la herramienta de último recurso cuando todo lo demás falla.
💡 Paginación: el offset penalty. OFFSET 50000 FETCH NEXT 20 debe leer y descartar 50,000 filas antes de retornar las 20. Con keyset pagination (WHERE Id > @lastId ORDER BY Id FETCH NEXT 20) es O(log n) sin importar qué página. La página 1,000 es igual de rápida que la página 1.
