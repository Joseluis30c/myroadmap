Cómo leer Execution Plans
SSMS Actual vs Estimated, operadores clave, warnings y qué buscar primero
⏱ ~5h
El Execution Plan es el radiografía de tu query. SQL Server decide cómo ejecutar tu query: qué índices usar, en qué orden unir tablas, cuándo paralelizar. El Estimated Plan es la predicción antes de ejecutar; el Actual Plan es lo que realmente pasó con estadísticas reales. Siempre usa Actual. Las diferencias entre estimated y actual rows revelan estadísticas obsoletas.

Operadores que debes conocer

Table Scan / Clustered Index Scan
Lee TODAS las filas de la tabla. En tablas grandes es devastador. El clásico "full scan".
Crear un índice en la columna del WHERE
Key Lookup (Bookmark Lookup)
Encontró la fila con un non-clustered index pero necesita ir a la tabla principal por más columnas.
Convertir a Covering Index con INCLUDE
Hash Match / Sort
Requiere memoria para operar. Si derrama a disco (Spill to TempDB) es muy lento.
Revisar estadísticas, índices, orden de joins
Index Seek
Navega directamente a las filas que necesita usando el B-tree. O(log n). Lo que quieres ver.
✓ Perfecto. El índice está siendo usado correctamente
Nested Loops (Index Seek)
JOIN eficiente cuando la tabla interior es pequeña y tiene índice. Muy rápido.
✓ Óptimo para joins con tablas pequeñas
Parallelism (Gather Streams)
SQL Server paralelizó la query. Puede ser bueno o señal de que algo no tiene índice.
Investigar por qué eligió paralelismo

Si tu condición envuelve la columna en una función o conversión implícita, SQL Server no puede usar el índice y hace un Scan completo. Clustered Index — Diseño correcto
Elección de la clave, fragmentación, GUID vs INT, fill factor y estadísticas
⏱ ~5h
La elección del Clustered Index define el rendimiento físico de tu tabla. Una buena clave clustered es: narrow (pocos bytes), unique, not null y ever-increasing (INT IDENTITY, no GUID). Un GUID como clustered key causa fragmentación masiva porque inserta filas en posiciones aleatorias del B-tree, forzando page splits constantemente.
⚠️ El GUID como Primary Key / Clustered Index es el error más común y costoso: Genera hasta 99% de fragmentación en tablas con muchos inserts. Usa INT IDENTITY o BIGINT IDENTITY como PK. Si necesitas un GUID como identificador único externo, ponlo como columna separada con un non-clustered index UNIQUE — no como clustered.

Non-Clustered + Covering Indexes
Composite indexes, columnas INCLUDE, filtered indexes y cuándo un índice cuesta más de lo que ayuda
⏱ ~5h
Covering Index = el índice tiene TODO lo que la query necesita. Cuando SQL Server usa un non-clustered index pero necesita columnas que no están en él, hace un Key Lookup — va a buscar esas columnas a la tabla principal, fila por fila. Agregar esas columnas en la cláusula INCLUDE del índice las pone en las hojas, eliminando el Key Lookup completamente.

📋 Anatomía de un índice y cuándo usar cada parte | IX_Orders_CustomerId_Date

Parte del índice Columnas Función Cuándo poner una columna aquí
KEY COLUMNS CustomerId, OrderDate Definen el orden del B-tree. Se usan en WHERE, JOIN, ORDER BY Columnas en cláusulas WHERE o JOIN. El orden importa — mayor selectividad primero.
INCLUDE COLUMNS Total, Status, ProductId Solo en las hojas. No ordenadas. Eliminan Key Lookups. Columnas que el SELECT necesita pero no son parte del criterio de búsqueda. Sin overhead de ordenamiento.
FILTER WHERE Status = 'Active' Filtered Index: solo indexa filas que cumplan el filtro Cuando consultas siempre el mismo subconjunto (activos, no eliminados, etc.). Índice más pequeño y rápido.

Optimización de Queries Reales
Workflow completo: encontrar la query lenta, leer el plan, crear el índice correcto y medir la mejora
⏱ ~5h
El workflow de optimización en 5 pasos: 1) Identificar la query más costosa con DMVs → 2) Obtener el Actual Execution Plan → 3) Analizar el operador más costoso (% más alto) → 4) Crear el índice recomendado o reescribir la query → 5) Medir logical reads antes vs después con SET STATISTICS IO ON. Un índice bien diseñado puede reducir de 50,000 a 3 logical reads.
💡 Regla de Brent Ozar: "If your query is slow, it's either the query or the indexes." Antes de cambiar el servidor, agregar RAM o acusar a la red — lee el execution plan. El 95% de los problemas de performance se resuelven en el SQL.
