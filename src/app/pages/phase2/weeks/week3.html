1
ACID y Manejo Correcto de Transacciones
BEGIN/COMMIT/ROLLBACK, TRY/CATCH, savepoints, transacciones anidadas y el patrón correcto en C# con EF Core
⏱ ~5h
Una transacción es una unidad de trabajo que debe completarse en su totalidad o no ejecutarse en absoluto. ACID garantiza: Atomicity (todo o nada), Consistency (estado válido antes y después), Isolation (las transacciones concurrentes no se interfieren), Durability (los cambios committed sobreviven a fallos). El 99% de los bugs de transacciones vienen de manejo incorrecto de errores: olvidar el ROLLBACK en el catch.
⚠️ El error más costoso: una transacción abierta sin COMMIT ni ROLLBACK bloquea todos los recursos que tocó. Si una excepción escapa sin el ROLLBACK correspondiente, tendrás bloqueos que crecen hasta paralizar el servidor. Siempre envuelve transacciones en TRY/CATCH con ROLLBACK en el CATCH.
❌ Patrón incorrecto
Sin TRY/CATCH. Si el INSERT falla, la transacción queda abierta bloqueando recursos. El ROLLBACK nunca se ejecuta.

BEGIN TRAN → INSERT → ❌ excepción → transacción abierta para siempre.
✓ Patrón correcto
TRY/CATCH con verificación de transacción activa antes del ROLLBACK. XACT_STATE() verifica si hay una transacción abierta y si es rollbackable.
⚡ C# + EF Core
using var tx = await context.Database.BeginTransactionAsync() con await tx.CommitAsync() en el try y await tx.RollbackAsync() en el catch.

2
Isolation Levels
Los 5 niveles, los fenómenos que cada uno permite, y por qué SNAPSHOT es la opción correcta para la mayoría de las aplicaciones
⏱ ~5h
El isolation level define qué anomalías de concurrencia son posibles en tu base de datos. Más isolation = menos anomalías = más locks = menos concurrencia. El default de SQL Server es READ COMMITTED — protege contra lecturas sucias pero permite non-repeatable reads. READ COMMITTED SNAPSHOT (RCSI) es una variante basada en versiones de fila que elimina bloqueos de lectura sin sacrificar consistencia. Es el que deberías usar por defecto en aplicaciones OLTP.

⚡ READ UNCOMMITTED / NOLOCK: el hint más peligroso del SQL. Permite leer datos que aún no fueron committed — puedes leer una fila que luego se hace ROLLBACK (dato fantasma) o una fila a la mitad de ser actualizada. NUNCA uses NOLOCK en datos financieros, inventario o cualquier dato donde la exactitud importe. "Más rápido pero incorrecto" no es una opción válida.
