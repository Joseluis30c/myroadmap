ACID y Manejo Correcto de Transacciones
BEGIN/COMMIT/ROLLBACK, TRY/CATCH, savepoints, transacciones anidadas y el patrón correcto en C# con EF Core
⏱ ~5h
Una transacción es una unidad de trabajo que debe completarse en su totalidad o no ejecutarse en absoluto. ACID garantiza: Atomicity (todo o nada), Consistency (estado válido antes y después), Isolation (las transacciones concurrentes no se interfieren), Durability (los cambios committed sobreviven a fallos). El 99% de los bugs de transacciones vienen de manejo incorrecto de errores: olvidar el ROLLBACK en el catch.
⚠️ El error más costoso: una transacción abierta sin COMMIT ni ROLLBACK bloquea todos los recursos que tocó. Si una excepción escapa sin el ROLLBACK correspondiente, tendrás bloqueos que crecen hasta paralizar el servidor. Siempre envuelve transacciones en TRY/CATCH con ROLLBACK en el CATCH.
❌ Patrón incorrecto
Sin TRY/CATCH. Si el INSERT falla, la transacción queda abierta bloqueando recursos. El ROLLBACK nunca se ejecuta.

BEGIN TRAN → INSERT → ❌ excepción → transacción abierta para siempre.
✓ Patrón correcto
TRY/CATCH con verificación de transacción activa antes del ROLLBACK. XACT_STATE() verifica si hay una transacción abierta y si es rollbackable.
⚡ C# + EF Core
using var tx = await context.Database.BeginTransactionAsync() con await tx.CommitAsync() en el try y await tx.RollbackAsync() en el catch.

Isolation Levels
Los 5 niveles, los fenómenos que cada uno permite, y por qué SNAPSHOT es la opción correcta para la mayoría de las aplicaciones
⏱ ~5h
El isolation level define qué anomalías de concurrencia son posibles en tu base de datos. Más isolation = menos anomalías = más locks = menos concurrencia. El default de SQL Server es READ COMMITTED — protege contra lecturas sucias pero permite non-repeatable reads. READ COMMITTED SNAPSHOT (RCSI) es una variante basada en versiones de fila que elimina bloqueos de lectura sin sacrificar consistencia. Es el que deberías usar por defecto en aplicaciones OLTP.

⚡ READ UNCOMMITTED / NOLOCK: el hint más peligroso del SQL. Permite leer datos que aún no fueron committed — puedes leer una fila que luego se hace ROLLBACK (dato fantasma) o una fila a la mitad de ser actualizada. NUNCA uses NOLOCK en datos financieros, inventario o cualquier dato donde la exactitud importe. "Más rápido pero incorrecto" no es una opción válida.

Locks — Cómo SQL Server controla el acceso
Tipos de lock (S, X, U, IS, IX), compatibilidad, escalada, blocking chains y cómo diagnosticarlos
⏱ ~5h
SQL Server usa locks para garantizar el aislamiento entre transacciones. Cuando una transacción lee datos, adquiere un Shared lock (S). Cuando modifica, necesita un Exclusive lock (X). Los locks son incompatibles: no puedes obtener un X si alguien tiene un S, y viceversa. El blocking ocurre cuando una sesión espera que otra libere su lock. El blocking no es un error — es el mecanismo correcto funcionando. El problema es cuando dura demasiado.

Tipos de lock en SQL Server

S — Shared
Para lecturas. Múltiples sesiones pueden tener S al mismo tiempo. Compatible con otros S, incompatible con X.
SELECT sin WITH(NOLOCK)
X — Exclusive
Para escrituras. Solo una sesión puede tener X. Incompatible con S y con otros X. Se mantiene hasta COMMIT/ROLLBACK.
INSERT / UPDATE / DELETE
U — Update
Previene deadlocks en patrones read-modify-write. Compatible con S pero solo uno puede existir. Se convierte a X antes de escribir.
UPDATE (fase de búsqueda)
IS — Intent Shared
Lock jerárquico en la tabla/página que indica que se van a tomar S locks en filas. Permite saber si una tabla está siendo leída.
Nivel tabla/página
IX — Intent Exclusive
Indica que se van a tomar X locks en filas. El motor verifica IX a nivel tabla antes de permitir locks completos.
Nivel tabla antes de X
Sch-S / Sch-M
Schema Stability / Modify. Se adquiere al ejecutar queries (Sch-S) o al modificar el esquema (ALTER TABLE). El Sch-M bloquea todo.
DDL / Compilación

Deadlocks — Diagnóstico y Prevención
Por qué ocurren, Deadlock Graph con Extended Events, los 4 patrones más comunes y cómo eliminarlos
⏱ ~5h
Un deadlock ocurre cuando dos sesiones se esperan mutuamente y ninguna puede continuar. Sesión A tiene lock sobre Recurso 1 y quiere Recurso 2. Sesión B tiene lock sobre Recurso 2 y quiere Recurso 1. SQL Server detecta automáticamente el ciclo y elige a una de las sesiones como "víctima" — la mata con error 1205, haciendo ROLLBACK de su transacción. La otra sesión continúa. El error 1205 en tu aplicación significa que hubo un deadlock y debes hacer retry.

🔍 El flujo de profiling profesional

SPID 52
HOLDS X lock en Orders(Id=1)
⟶ WANTS X lock en Inventory(Id=1)
⟵ DEADLOCK ⟶
SPID 53
HOLDS X lock en Inventory(Id=1)
⟶ WANTS X lock en Orders(Id=1)

SQL Server elige víctima: SPID 53 recibe error 1205, su transacción hace ROLLBACK. SPID 52 continúa y hace COMMIT. Tu aplicación debe tener retry logic para el 1205.

Simular Escenarios Concurrentes
Lost update, optimistic vs pessimistic concurrency, SELECT FOR UPDATE, UPDLOCK hints y el entregable final
⏱ ~5h
El problema del Lost Update: Dos usuarios leen el mismo stock (100 unidades), ambos restan 10, ambos guardan 90. El resultado correcto sería 80. Este es el escenario concurrente más común y tiene dos soluciones: Pessimistic concurrency (bloquear la fila al leer con UPDLOCK para que nadie más la modifique) o Optimistic concurrency (no bloquear al leer, pero verificar antes de guardar que nadie cambió la fila — usando un rowversion/timestamp).
