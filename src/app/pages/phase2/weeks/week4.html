1
Identificar Cuellos de Botella
Wait Stats, sp_BlitzCache, Query Store y DMVs — encontrar las queries problemáticas sin adivinar
⏱ ~5h
Nunca optimices sin datos. El 90% de los problemas de performance están en el 10% del código. Las Wait Statistics son el punto de partida: dicen exactamente qué espera SQL Server — si espera I/O (PAGEIOLATCH), CPU, locks (LCK_M_X) o red (ASYNC_NETWORK_IO). Con ese diagnóstico, vas a sp_BlitzCache para encontrar las queries exactas que causan esas esperas. Solo entonces abres el execution plan.

🗺 Workflow de identificación — en este orden exacto

1 Wait Statistics — ¿Qué espera el servidor?
Ejecuta sys.dm_os_wait_stats para ver los top wait types. PAGEIOLATCH_SH = falta memoria o índices. LCK_M_X = bloqueos. CXPACKET = paralelismo excesivo. ASYNC_NETWORK_IO = la app lee filas demasiado lento. El wait type te dice el tipo de problema antes de ver una sola query.
Herramientasys.dm_os_wait_stats
2 sp_BlitzCache — Top queries por impacto
Ejecuta sp_BlitzCache @SortOrder = 'reads' para ver las queries ordenadas por lecturas lógicas. Luego 'cpu', 'duration' y 'executions'. Una query que ejecuta 10,000 veces con lecturas moderadas puede ser más impactante que una que ejecuta 1 vez con lecturas enormes.
Herramientasp_BlitzCache
3 Query Store — Historial y regresiones
En SSMS: Query Store → Top Resource Consuming Queries. Filtra por las últimas 24h. Permite comparar el plan de hoy con el de ayer — si la misma query tiene planes distintos, hay parameter sniffing o estadísticas obsoletas. También muestra queries con plan regression automáticamente.
HerramientaQuery Store SSMS
4 Priorizar por impacto × frecuencia
Score = avg_logical_reads × execution_count. Una query con 1,000 reads que ejecuta 50,000 veces/día tiene score 50M — mucho más impactante que una con 500,000 reads que ejecuta 2 veces/semana. Optimiza por score total, no por la query que "se ve más fea".

3
Refactorizar Stored Procedures Complejos
Las 8 transformaciones más impactantes: cursores→sets, temp tables, índices cubrientes, parameter sniffing, reescritura de JOINs
⏱ ~5h
Un SP complejo tiene capas de problemas apilados. El orden correcto de ataque: primero los índices (mayor impacto, menor riesgo), luego reescritura de queries (eliminar scans, cursores, subconsultas correlacionadas), luego parameter sniffing si el plan varía por parámetros, y por último arquitectura (temp tables intermedias, dividir el SP en partes). Un cambio a la vez — mide después de cada uno.

Anatomía de un SP problemático — diagnóstico por zonas | usp_GenerateMonthlyReport

🔴 Zona 1 — Cursor Loop
Itera fila por fila con CURSOR FOR SELECT. Ejecuta N queries en lugar de 1. Impacto: 100x–1000x más lento que la versión set-based.
🔴 Zona 2 — Subconsulta N+1
Subconsulta correlacionada en el SELECT que ejecuta una vez por cada fila del resultado exterior. Con 10,000 clientes = 10,000 queries adicionales.
🟡 Zona 3 — Table Variable
Usa @TableVariable con JOIN posterior. Sin estadísticas → optimizer estima 1 fila → plan incorrecto. Reemplazar con #TempTable.
🟡 Zona 4 — Función en WHERE
Condición WHERE YEAR(OrderDate) = 2024. Non-SARGable → Clustered Index Scan en lugar de Seek. Reescribir con rango explícito.
🟡 Zona 5 — Parameter Sniffing
Plan compilado para @CustomerId=1 (VIP con 50k órdenes) usado para @CustomerId=999 (1 orden). Nested Loops × 50k cuando debería ser 1 seek.
🟢 Zona 6 — Oportunidad de índice
Missing Index en la tabla Orders para (CustomerId, OrderDate) INCLUDE (Total, Status). Elimina el Clustered Index Scan del 78% del costo.

Medir Después — Validar y Comparar
Tabla antes/después, comparar planes XML, regresar si empeora, verificar que no rompiste nada
⏱ ~5h
La medición post-optimización es tan importante como el cambio mismo. Debes verificar tres cosas: que los logical reads bajaron (menos I/O), que el elapsed time bajó (más rápido para el usuario), y que los resultados son idénticos (mismas filas, mismo orden si aplica). Un cambio que mejora 10x la performance pero retorna datos incorrectos no es una optimización — es un bug.
💡 Comparar planes XML con SentryOne Plan Explorer: guarda el plan XML antes y después (clic derecho en el plan → Save Execution Plan As), luego ábrelos en Plan Explorer y usa la vista "Compare Plans". Verás exactamente qué operadores cambiaron, qué índice se está usando ahora y dónde fue la mejora de costo.

Documento Técnico de Mejoras
Estructura del reporte profesional, cómo comunicar mejoras a stakeholders técnicos y no técnicos, y el entregable final
⏱ ~5h
Un documento técnico de performance tiene dos audiencias simultáneas: los desarrolladores (que necesitan los detalles técnicos para reproducir y mantener las mejoras) y los stakeholders (que necesitan entender el impacto de negocio en términos de tiempo, costo y experiencia de usuario). La estructura correcta satisface a ambos: resumen ejecutivo primero, detalles técnicos después.

Documento Técnico: Informe de Optimización de Performance — [Proyecto]

1. Resumen Ejecutivo
3 stored procedures identificados como cuellos de botella críticos. Después de la optimización, el tiempo promedio de respuesta del endpoint /api/orders se redujo de 2,840ms a 198ms (14.3x). Reducción estimada en costo de servidor: 40% de CPU en horas pico.
2. Metodología de Diagnóstico
Herramientas usadas: sys.dm_os_wait_stats (Wait Stats baseline), sp_BlitzCache (top queries por reads×executions), Query Store (historial de planes), SET STATISTICS IO/TIME (métricas por query). Criterio de priorización: score = avg_logical_reads × execution_count/día.
3. Hallazgos por Stored Procedure
Para cada SP: problema identificado (operador en el plan), causa raíz (missing index / cursor / non-SARGable / parameter sniffing), cambio aplicado (código antes/después), métricas comparativas (tabla con logical reads, elapsed time, plan antes vs plan después).
4. Cambios Aplicados
Scripts DDL de los índices creados (con justificación de cada columna), código T-SQL de los SPs refactorizados, configuración de RCSI si se habilitó, actualización de estadísticas con FULLSCAN.
5. Verificación y Regresión
Comparación EXCEPT de resultados old vs new (0 diferencias). Unit tests ejecutados antes/después. Métricas en entorno QA que replican producción.
6. Próximos Pasos Recomendados
3 queries adicionales identificadas con score medio. Recomendación de mantenimiento de estadísticas automático. Habilitar Query Store en producción para monitoreo continuo.

🎯 Entregable — Documento técnico con mejoras de performance

El entregable de esta semana es un documento técnico completo que demuestra el ciclo completo de optimización aplicado a casos reales de tu BD o del StackOverflow DB de Brent Ozar:

  • Sección de diagnóstico: Wait Stats analizados + top 3 queries identificadas con sp_BlitzCache y justificación de la priorización
  • Tabla de baseline: logical reads, CPU time, elapsed time y plan XML guardado para cada query antes de cualquier cambio
  • Para cada query optimizada: código SQL antes vs después, justificación del cambio, y el execution plan antes/después como imagen o XML
  • Scripts DDL de todos los índices creados con comentarios que explican cada columna KEY y cada columna INCLUDE
  • Tabla comparativa final: logical reads, elapsed time y mejora multiplicativa (Nx) para las 3 queries
  • Verificación de correctitud: resultado EXCEPT query confirma 0 diferencias entre versión original y optimizada
