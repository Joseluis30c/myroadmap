xUnit — Fundamentos
Facts, Theories, fixtures, ciclo de vida y FluentAssertions
⏱ ~4h
¿Por qué xUnit sobre NUnit? xUnit es el framework más moderno y el que usa el equipo de .NET internamente. No tiene [SetUp] ni [TearDown] — usa constructores y IDisposable, lo que es más limpio en OOP. [Fact] es un test simple; [Theory] con [InlineData] es un test parametrizado que corre N veces con datos distintos.

📋 Convenciones de naming — elige una y sé consistente:

Estilo 1 — El más popular
MethodName_StateUnderTest_
ExpectedBehavior
Create_WithEmptyName_
ThrowsDomainException
Estilo 2 — BDD-like
Should_ExpectedBehavior_
When_StateUnderTest
Should_ThrowException_
When_NameIsEmpty
Estilo 3 — Frase natural
Creating_product_with_
empty_name_fails
Muy legible, popular en equipos con BDD

Moq — Mocking de Dependencias
Mock, Setup, Verify, Returns, Callback y cuándo NO mockear
⏱ ~5h
¿Para qué sirve Moq? Cuando el servicio que pruebas tiene dependencias (un repositorio, un servicio de email), no quieres usar las implementaciones reales en los tests. Moq crea objetos falsos (mocks) que puedes controlar: "cuando alguien llame a GetByIdAsync, devuelve este producto". Así pruebas solo la lógica de tu servicio, no la del repositorio.
💡 Regla de oro del mocking: Solo mockea lo que cruzas un boundary arquitectónico: repositorios, servicios externos, email, etc. NO mockees clases concretas de tu propio dominio (Value Objects, Entities) — esas se usan directamente. Si sientes que necesitas mockear demasiado, probablemente tu diseño tiene demasiado acoplamiento.
⚠️ Moq v4.20+ cambió el modelo de licencia. Si trabajas en empresa, evalúa usar NSubstitute como alternativa (misma funcionalidad, licencia BSD, sintaxis igual de limpia). Ambos se cubren aquí.

Testing de Controllers
Testear action results, status codes, y cuándo usar integration tests en su lugar
⏱ ~5h
¿Vale la pena testear Controllers unitariamente? Hay debate en la comunidad. Los Controllers son delgados (thin controllers) — solo mapean Result a HTTP. Un unit test de Controller prueba básicamente if (result.IsSuccess) return Ok(). La mayoría del valor está en los tests de servicios. Aun así, vale la pena para cubrir todos los status codes y asegurarse que el mapping es correcto.

Cobertura de Código + Entregable Final
Coverlet, reportes HTML, qué medir y qué no, y alcanzar el 60%
⏱ ~5h
Cobertura no es calidad: 100% de cobertura con assertions vacías es inútil. El objetivo no es el número, es usar el número como indicador de huecos en tu suite de tests. Enfócate en cubrir los caminos de decisión (branch coverage), no solo las líneas.
💡 Tip para llegar al 60%: Domain Layer (entidades + value objects) es lo más fácil — 0 mocks, lógica pura, debería estar al 90%+ solo. Application Layer con Moq lleva el resto. Infrastructure no cuentes para el 60% — eso va con Integration Tests la semana que viene.

📊 ¿Qué significa cada nivel de cobertura?

Cobertura Interpretación Típico de
0-40%
 Sin tests reales o muy básicos Proyectos legacy, prototipos
40-60%
 Tests básicos cubriendo happy paths Proyectos en crecimiento — tu meta esta semana
60-80%
 Happy + error paths cubiertos Estándar profesional mínimo
80%+
 Cobertura robusta de la lógica crítica Librerías, código de misión crítica

💻 Comandos — Generar reporte de cobertura HTML

Paso a paso para ver tu cobertura en el browser

# 1. Instalar ReportGenerator como herramienta global
dotnet tool install -g dotnet-reportgenerator-globaltool

# 2. Ejecutar tests con cobertura
dotnet test --collect:"XPlat Code Coverage" \
            --results-directory ./coverage

# 3. Generar reporte HTML
reportgenerator \
  -reports:"./coverage/**/coverage.cobertura.xml" \
  -targetdir:"./coverage/report" \
  -reporttypes:Html

# 4. Abrir el reporte
open ./coverage/report/index.html   # macOS
start ./coverage/report/index.html  # Windows

# Alternativa: ver solo el summary en consola
dotnet test /p:CollectCoverage=true /p:CoverletOutputFormat=cobertura \
            /p:Threshold=60 /p:ThresholdType=line

🎯 Entregable — 60% de Cobertura en el Proyecto

Para aprobar la semana debes poder ejecutar dotnet test y el reporte mostrar:

  • Domain Layer: 90%+ de cobertura (Entities, Value Objects, reglas de negocio)
  • Application Layer: 70%+ de cobertura (todos los casos de uso, happy y error paths)
  • Al menos 20 tests en total, todos pasando en verde con dotnet test
  • FluentValidation validators testeados con TestValidate de la Semana 3
  • Mocks verificados con Times.Once y Times.Never en tests críticos
  • Reporte HTML de Coverlet generado y cobertura global ≥ 60%
  • Builder pattern para test data (al menos un Builder reutilizable)
  • Naming consistente en todos los tests (elige uno de los 3 estilos y úsalo siempre)
🔗 Conexión con Semanas 1–3: Los validators de FluentValidation (Semana 3) se testean con validator.TestValidate(request) — sin levantar HTTP. El Result Pattern hace tus assertions más simples: result.IsSuccess.Should().BeTrue(). Los IExceptionHandler (Semana 3) se prueban mejor con Integration Tests (Semana 5) — esta semana solo unit tests.
