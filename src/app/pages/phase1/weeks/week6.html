Con async: los mismos 2 threads manejan 8–10 requests simultáneas. Mientras espera la DB, el thread atiende otra request. El kernel notifica cuando el I/O termina.

💡 La regla clave: async/await no hace que una operación termine más rápido. Hace que el thread no se quede bloqueado esperando. El beneficio es en throughput (cuántas requests por segundo), no en la latencia de una request individual.

Async/Await Profundo
State machines, SynchronizationContext, ConfigureAwait y cómo realmente funciona el compilador
⏱ ~4h
Lo que hace el compilador con async/await: Cuando escribes await, el compilador transforma tu método en una state machine. El método se pausa, devuelve control al caller, y cuando el I/O termina, el kernel notifica al ThreadPool que retome la ejecución desde donde quedó. No hay thread bloqueado durante la espera — es cooperativo, no preemptivo.
💡 ConfigureAwait(false): En código de librería o Application/Infrastructure Layer, usa siempre await algo.ConfigureAwait(false). Le dices al runtime que no necesitas volver al contexto original (UI thread o ASP.NET request context). Mejora performance y evita deadlocks en contextos síncronos.

Task vs Thread — El modelo de concurrencia de .NET
ThreadPool, Parallel, IAsyncEnumerable y cuándo usar cada primitiva
⏱ ~5h
La confusión más común: Task NO es un Thread. Un Task es una promesa de un resultado futuro. Puede ejecutarse en el ThreadPool, de manera síncrona, o nunca crear un thread (I/O bound). Thread es un hilo del sistema operativo — costoso (1MB de stack), lento de crear y destruir. Usa Tasks siempre; crea Threads solo en casos muy específicos (CPU-bound intensivo y de larga duración).

📋 ¿Qué primitiva usar para cada escenario?

Escenario Usar Por qué
Request a BD, HTTP, File async/await Task ✅ I/O-bound: libera el thread mientras espera
Múltiples operaciones independientes Task.WhenAll ✅ Paralelismo real sin bloquear
Procesar colección grande en memoria Parallel.ForEach ⚠️ CPU-bound: usa múltiples cores, cuidado con ASP.NET
Streaming de resultados de BD IAsyncEnumerable ✅ No carga todo en memoria, emite de a poco
Background job de larga duración IHostedService ✅ No bloquea el ThreadPool de Kestrel
CPU-bound intensivo (video, crypto) Task.Run ⚠️ Mueve al ThreadPool, úsalo con moderación en APIs
Crear un Thread manualmente new Thread() ❌ Casi nunca en .NET moderno. Usa Task.Run o IHostedService

Anti-patrones Async — Los errores que matan el performance
async void, .Result/.Wait(), fire-and-forget mal hecho y sync-over-async
⏱ ~4h
El anti-patrón más peligroso: .Result y .Wait(): Bloquean el thread actual esperando que la Task complete. En ASP.NET Core, si el SynchronizationContext está configurado, esto provoca un deadlock garantizado — el thread espera la Task, y la Task espera ese mismo thread para continuar. Tu endpoint cuelga para siempre.
💀
.Result / .Wait() en ASP.NET
Bloquea el thread. En escenarios con SynchronizationContext causa deadlock garantizado.
Usa await siempre. Todo el camino hacia arriba.
🔥
async void (excepto event handlers)
Las excepciones se pierden silenciosamente. No se puede hacer await. El llamador no sabe cuándo termina.
Retorna Task siempre. async void solo en event handlers de UI.
🐌
await dentro de un loop
N llamadas secuenciales cuando podrían ser paralelas. 10 llamadas de 100ms = 1000ms en vez de 100ms.
Task.WhenAll para operaciones independientes.
🪤
Task.Run para I/O bound
Task.Run toma un thread del ThreadPool innecesariamente para operaciones que no necesitan CPU.
Solo Task.Run para trabajo CPU-bound real.
🧨
Fire-and-forget sin manejo
_ = DoSomethingAsync() lanza una Task no observada. Las excepciones se pierden.
Usa IHostedService o BackgroundService para background work.
⛔
No propagar CancellationToken
Si el cliente cancela la request, tu código sigue ejecutando queries a la BD innecesariamente.
Acepta y propaga CancellationToken en todos los métodos async.

⚙️ Parámetros de Kestrel más importantes para performance

MaxConcurrentConnections
Máximo de conexiones TCP simultáneas
Default: null (ilimitado)
MaxRequestBodySize
Tamaño máximo del body en bytes
Default: 30,000,000 (30MB)
KeepAliveTimeout
Tiempo máximo de keep-alive de conexión
Default: 130 segundos
RequestHeadersTimeout
Tiempo máximo para recibir headers
Default: 30 segundos
Http2.MaxStreamsPerConnection
Streams simultáneos por conexión HTTP/2
Default: 100
AllowSynchronousIO
Permite I/O síncrono (nunca en producción)
Default: false

Optimización de Endpoints
IMemoryCache, Span<T>, Output Caching, pooling y técnicas de optimización concretas
⏱ ~5h
Las optimizaciones de mayor impacto, en orden: 1) Reducir trips a la BD (caché + queries eficientes) → 2) Async I/O correcto → 3) Compresión de responses → 4) Output Caching para endpoints repetidos → 5) Allocations (Span, ArrayPool) para hot paths muy específicos. Empieza siempre por el 1.
💡 EF Core y performance: El 80% de los problemas de performance en APIs .NET vienen de queries ineficientes a la BD. AsNoTracking() en queries de solo lectura, proyecciones con Select() en lugar de cargar toda la entidad, e índices correctos en la BD resuelven la mayoría de los casos antes de necesitar caché.

🎯 Entregable — Optimización de Endpoints Lentos

Elige 2 endpoints de tu API y optimízalos demostrando la mejora con números:

  • Benchmark before/after con BenchmarkDotNet para al menos 2 endpoints
  • Cero instancias de .Result, .Wait() o async void en todo el proyecto
  • Todos los reads con AsNoTracking() + Select() (proyecciones en lugar de entidades completas)
  • Task.WhenAll en cualquier lugar donde había await en loop con operaciones independientes
  • CancellationToken propagado desde los controllers hasta los repositorios
  • Output Caching en al menos un endpoint de lectura frecuente
  • Response Compression (Brotli + Gzip) habilitado en Kestrel
  • Reporte de dotnet-trace o dotnet-counters mostrando métricas de la API
