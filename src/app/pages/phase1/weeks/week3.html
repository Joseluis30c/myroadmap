📋 ProblemDetails – RFC 9457 (el estándar que debes usar)

Todos los errores de tu API deben responder con este formato. Es el estándar de la industria y .NET lo soporta nativamente desde .NET 7.

"type"
URI que identifica el tipo de error. Ej: "https://tools.ietf.org/html/rfc9110#section-15.5.5"
"title"
Descripción corta legible. Ej: "Validation Error" o "Not Found"
"status"
HTTP status code. Ej: 400, 404, 422, 500
"detail"
Explicación específica del error en esta request concreta
"instance"
URI de la request que causó el error. Ej: "/products/abc123"
"errors"
Extensión custom: lista de errores de validación por campo

FluentValidation Avanzado + Pipeline de Validación
Custom validators, validación async, When/Unless, RuleSets y Validation Behavior
⏱ ~5h
Validation Behavior (el patrón pro): En lugar de llamar al validator manualmente en cada endpoint, creas un ValidationBehavior<TRequest,TResponse> que intercepta todos los Commands/Queries automáticamente. Si falla, lanza una excepción antes de llegar al Application Service. Cero boilerplate.
💡 Tip: Los validators async son para validaciones que requieren consultar la BD, como verificar si un email ya existe. Úsalos con cuidado — agregan latencia. Para la mayoría de reglas, los validators síncronos son suficientes.

Manejo Global de Excepciones
IExceptionHandler, ProblemDetails, mapeo de errores y middleware de excepciones
⏱ ~5h
El principio: Ningún try/catch en los Controllers. Los errores se propagan hacia arriba y un handler global los captura, los mapea a un formato estándar (ProblemDetails) y retorna la respuesta correcta. Los controllers quedan limpios.
⚠️ Anti-patrón frecuente: Tener try/catch en cada action del Controller que retorna Ok() o BadRequest() con mensajes inconsistentes. El resultado: 10 formatos de error diferentes en la misma API.

API Response Pattern
Result pattern, respuestas consistentes, envelopes y cuándo NO usar wrapper
⏱ ~4–5h
Result Pattern vs Excepciones: Usar excepciones para flujo de control normal (NotFoundException en un GET) es un anti-patrón. El Result Pattern modela explícitamente el éxito y el fracaso como valores de retorno: Result<Product> puede ser un éxito con el producto o un error con el motivo. Hace el código más predecible y testeable.

Buenas Prácticas REST + API Entregable
HTTP status codes correctos, naming conventions, versioning básico e integrar todo
⏱ ~5h
REST no es solo usar HTTP: Es usar HTTP correctamente. Los recursos son sustantivos (no verbos), los status codes tienen significado preciso, los headers comunican metadatos y el contenido de los errores sigue un estándar. Una API REST bien diseñada es autodocumentada.

📋 Códigos HTTP que más usarás y cuándo

CódigoNombreCuándo usarlo
200OKGET exitoso, PUT/PATCH exitoso con body de respuesta
201CreatedPOST que crea un recurso. Incluir header Location
204No ContentDELETE exitoso, PUT/PATCH sin body de respuesta
400Bad RequestErrores de validación de input (FluentValidation)
401UnauthorizedSin autenticación o token inválido
403ForbiddenAutenticado pero sin permiso para ese recurso
404Not FoundRecurso no encontrado. No para "lista vacía"
409ConflictEstado conflictivo: email duplicado, versión optimistic lock
422Unprocessable EntityInput válido sintácticamente pero inválido semánticamente
500Internal Server ErrorError inesperado del servidor. Nunca exponer detalles en prod

🎯 Entregable — API con Validaciones y Manejo Global de Errores

Construida sobre la Clean Architecture de la Semana 2, tu API debe ahora también cumplir:

  • FluentValidation con validators en el Application Layer, auto-registrados por DI
  • Todos los errores en formato ProblemDetails — cero formatos de error custom ad-hoc
  • IExceptionHandler para ValidationException, NotFoundException y un handler global fallback
  • Result<T> implementado en el Application Layer para errores de negocio esperados
  • Controllers sin try/catch, con status codes HTTP correctos por cada operación
  • Al menos 5 unit tests de validators con TestValidate (sin necesidad de HTTP)
  • URLs REST correctas: /api/v1/products, /api/v1/products/{{'{'}}id{{'}'}}
  • API Versioning configurado aunque sea con v1 por ahora
🔗 Conexión con Semanas 1 y 2: El middleware de la Semana 1 convive con UseExceptionHandler() — el handler de excepciones va antes en el pipeline. Los validators de FluentValidation viven en el Application project de tu Clean Architecture. El Result<T> vive en el Domain. El IExceptionHandler vive en la capa Api. Todo encaja en la estructura de la Semana 2.
