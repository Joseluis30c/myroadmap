🗺 Flujo completo de autenticación con JWT + Refresh Token

POST /auth/login — Usuario envía email + password
La API valida credenciales contra la BD (hash bcrypt). Si son válidas, genera tokens.
POST
API genera Access Token (15 min) + Refresh Token (7 días)
Access Token: JWT firmado con claims del usuario. Refresh Token: GUID opaco guardado en BD + cookie HttpOnly.
JWT
Cliente usa el Access Token en el header Authorization: Bearer {{'{'}}token{{'}'}}
Cada request protegida envía el JWT. El servidor lo valida sin consultar la BD (stateless).
Bearer
Middleware JWT valida firma, expiración y claims automáticamente
Si el token es válido, el ClaimsPrincipal se inyecta en HttpContext.User. Si no, 401.
401 / OK
Access Token expira → POST /auth/refresh con el Refresh Token
La API valida el refresh token en BD, lo rota (invalida el anterior, genera uno nuevo) y emite nuevo Access Token.
Rotate
POST /auth/logout → Revoca el Refresh Token en BD
El Access Token sigue válido hasta que expire (corta duración lo hace aceptable). El refresh token queda inutilizable.
Revoke

⚠️ Los 6 errores de seguridad JWT más comunes — aprende a evitarlos desde el día 1

Access Token de larga duración
Expiración de 24h o más hace que tokens robados sean útiles por mucho tiempo.
✅ Fix: Máximo 15 min. Usa Refresh Tokens.
Secret Key débil o en appsettings
Clave corta o commiteada al repositorio es catastrófico.
✅ Fix: Mínimo 256 bits. Usar User Secrets / Environment Variables.
Datos sensibles en el payload
El payload es Base64, cualquiera lo puede leer. No es cifrado.
✅ Fix: Solo IDs, roles y claims mínimos necesarios.
Refresh Token en localStorage
Vulnerable a ataques XSS. El script malicioso lo roba fácilmente.
✅ Fix: Refresh Token en cookie HttpOnly + Secure.
Sin rotación de Refresh Tokens
Si el refresh token se roba, el atacante puede renovar indefinidamente.
✅ Fix: Rotar (invalidar + generar nuevo) en cada uso.
Sin validar el algoritmo
El ataque "alg:none" permite forjar tokens sin firma válida.
✅ Fix: Especificar explícitamente el algoritmo válido. Nunca "none".

Roles y Claims
Role-based y Policy-based authorization, claims personalizados y ClaimsPrincipal
⏱ ~4–5h
Roles vs Policies vs Claims: Roles son simples grupos ("Admin", "User"). Claims son pares clave-valor sobre el usuario ("department: engineering", "subscription: premium"). Policies son reglas que combinan claims, roles y lógica custom. En APIs modernas, las policies son más flexibles y recomendadas sobre los roles solos.

📋 Claims estándar (RFC 7519) que deberías conocer

Claim Nombre Uso
sub Subject ID único del usuario — la identidad principal
iss Issuer Quién emitió el token — tu API URL
aud Audience Para quién es el token — tu frontend URL
exp Expiration Unix timestamp de expiración
iat Issued At Cuándo fue emitido
jti JWT ID ID único del token — permite revocación y detección de replay
email Email Email del usuario (claim custom pero muy usado)
role Roles Rol(es) del usuario para autorización

Refresh Tokens
Implementación segura, rotación, almacenamiento en BD y revocación
⏱ ~5h
¿Por qué necesitas Refresh Tokens? Los Access Tokens duran 15 minutos. Sin refresh tokens, el usuario tendría que hacer login cada 15 minutos. El Refresh Token es un token opaco (GUID) de larga vida (7 días) que permite obtener nuevos Access Tokens sin pedir credenciales otra vez. Siempre se almacena hasheado en la BD y se rota en cada uso.
💡 Refresh Token Rotation: Cada vez que se usa un refresh token para obtener uno nuevo, el anterior se invalida inmediatamente. Si detectas que un refresh token ya usado intenta usarse de nuevo → alguien lo robó → invalida todos los tokens del usuario (Refresh Token Family Attack detection).

Protección de Endpoints
Authorize attributes, resource-based auth, Rate Limiting y CORS correcto
⏱ ~5h
Defense in depth: La seguridad de un endpoint no es solo [Authorize]. Es la combinación de autenticación JWT + autorización por roles/policies + rate limiting (evitar brute force) + CORS correctamente configurado (no AllowAnyOrigin) + validación de input (Semana 3). Cada capa protege contra un vector diferente.
⚠️ CORS no es seguridad: CORS solo controla qué orígenes del browser pueden hacer requests. No protege contra ataques server-to-server, curl, Postman ni ninguna herramienta que no sea un browser. CORS es para UX, no para seguridad del servidor.

🎯 Entregable — API con Autenticación Robusta

Tu API debe poder superar este checklist de seguridad:

  • POST /auth/login devuelve Access Token (15 min) + Refresh Token en cookie HttpOnly
  • POST /auth/refresh rota el Refresh Token y emite nuevo Access Token
  • POST /auth/logout revoca el Refresh Token y borra la cookie
  • Endpoints protegidos con [Authorize] retornan 401 sin token válido
  • Endpoints Admin retornan 403 si el usuario autenticado no tiene el rol
  • Rate limiting en /auth/login: 429 después del 5to intento en 1 minuto
  • CORS configurado aceptando solo el origen de tu frontend
  • Unit tests del JwtTokenService y del AuthService con Moq
🔗 Conexión con semanas anteriores: El JwtTokenService va en tu capa Infrastructure (Semana 2) e implementa una interfaz IJwtTokenService definida en Application. Los validators de FluentValidation (Semana 3) protegen el endpoint de login de inputs malformados. El IExceptionHandler (Semana 3) captura las excepciones de autenticación y las mapea a ProblemDetails con status 401/403. Los unit tests del AuthService (Semana 4) mockean IUserRepository, IJwtTokenService y IRefreshTokenRepository.
