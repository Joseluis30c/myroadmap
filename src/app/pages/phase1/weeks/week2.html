Principios de Clean Architecture
Dependency Rule, SOLID, por qué separar capas y qué problema resuelve
⏱ ~4–5h
La regla fundamental: Las dependencias solo pueden apuntar hacia adentro. El Domain no sabe que existe Infrastructure. La Application no sabe que existe la API. Esto se logra con interfaces: el Domain define IProductRepository, y la Infrastructure lo implementa.
Single Responsibility
Cada clase/módulo tiene una sola razón para cambiar. Un controller no contiene lógica de negocio.
Open/Closed
Abierto a extensión, cerrado a modificación. Agregar un nuevo repositorio sin tocar el caso de uso.
Dependency Inversion
Los módulos de alto nivel no dependen de los de bajo nivel. Ambos dependen de abstracciones (interfaces).
Interface Segregation
Interfaces pequeñas y específicas. IReadRepository separado de IWriteRepository.

Domain Layer — El núcleo
Entities, Value Objects, interfaces de repositorios, Domain Events y reglas de negocio
⏱ ~5h
Regla de oro del Domain: Este proyecto no referencia ningún NuGet externo (excepto abstracciones puras). No conoce Entity Framework, no conoce la base de datos, no conoce HTTP. Solo contiene la lógica de negocio pura en C#.
💡 Tip práctico: Si tu entidad Product tiene un precio que no puede ser negativo, esa validación va en el Domain, no en el Controller ni en un Validator de FluentValidation. La invariante de negocio vive donde vive el negocio.

Repository Pattern bien aplicado + API Entregable
Generic Repository, anti-patrones a evitar, y reestructurar tu API de la semana 1
⏱ ~5h
Repository Pattern bien aplicado: No es solo un wrapper de DbContext. El repositorio debe hablar el lenguaje del dominio: GetActiveProductsAsync(), no GetWhere(p => p.IsActive). El Generic Repository puede ser útil como base, pero los repositorios específicos deben tener métodos propios del dominio.
⚠️ Anti-patrón a evitar: Exponer IQueryable<T> desde el repositorio. Esto rompe la abstracción — el caller empieza a construir queries con lógica que debería estar en el repositorio o en el Domain.

🎯 Entregable – API Reestructurada con Clean Architecture

Tu API de la Semana 1 debe quedar reestructurada cumpliendo estos puntos:

  • 4 proyectos independientes: Domain, Application, Infrastructure, Api
  • Domain sin referencias a NuGets de infraestructura (0 dependencias externas)
  • Entidades con invariantes de negocio (precio ≥ 0, nombre requerido, etc.)
  • Al menos un Value Object implementado (ej: Money, Email, Address)
  • IProductRepository definido en Domain, implementado en Infrastructure
  • EF Core con IEntityTypeConfiguration por entidad y ValueObject conversions
  • Unit tests del Domain que no requieren base de datos ni mocks
  • Logging de la Semana 1 integrado en el Application Layer
